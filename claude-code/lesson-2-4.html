When Claude Code does it wrong

The best Claude Code users aren't the ones who get perfect prompts on the first go. They're the ones who iterate fastest. Here's how they do it — and why it's the skill that separates the two groups.

First output is rarely perfect. That's not a Claude problem — it's a reality of working with any capable assistant. You describe a task, they do their version of it, and then you shape the result through feedback. Humans do this. AI does this. The skill is in the feedback.

Most people, the first few weeks they use Claude Code, iterate badly. They say "that's wrong, try again" and get the same output. They rewrite their prompt from scratch and lose all the context. They start over four times before lunch. This lesson is about not being that person.

!
The common misconception

"If Claude got it wrong, I should rewrite the prompt better." Sometimes, yes. But most of the time, the faster fix is to tell Claude what specifically to change. The output in front of you already has most of what you wanted — you just need to nudge the parts that missed. Rewriting from scratch throws away all the useful context Claude has already built up.

Bad feedback vs. good feedback

The difference between someone who gets to a good result in two tries and someone who needs ten is almost entirely in how they phrase corrections. Here's the pattern, as starkly as I can show it:

Doesn't work

"That's wrong."

Claude has no idea which part is wrong. It'll guess — usually by rewriting the whole output with slightly different phrasing. The second attempt is often worse.

Works

"You processed the April PDFs but missed the three invoices without invoice numbers. Include those in the summary with 'no invoice number' in that column."

Claude knows exactly what to fix, what to leave alone, and how to handle the edge case. Single clean iteration to the right answer.

Notice what the good version has that the bad version doesn't:

  • A specific observation about what happened. "You processed the April PDFs but missed three without invoice numbers" — a factual description of the output, not a judgement of it.
  • A specific instruction about what to change. "Include those in the summary with 'no invoice number' in that column" — not "try again," but "here's what to do differently."
  • Implicit preservation of what already worked. By naming the change, you're also saying "the rest is fine." Claude won't accidentally rewrite the parts you were happy with.

This is almost identical to giving feedback to a junior colleague. "This isn't quite right" is useless feedback to them too. "The first half is perfect; the conclusion doesn't mention the follow-up call we agreed" is actionable. Same with Claude.

Diagnose the failure using IWO

In the last lesson, we looked at prompts as three parts: inputs, work, and outputs. That same framing is the fastest way to diagnose which part of an output went wrong — and therefore what specific feedback to give.

What you see
Which part failed
How to fix it
Missed files, included the wrong files, or processed files you didn't mean
Inputs
Name the exact folder, file pattern, or file list. "Read only the .pdf files in /inputs/, not the .docx ones."
Did the wrong thing with the right files — categorised badly, summarised incorrectly, extracted the wrong fields
Work
Describe the work more specifically. "For each invoice, extract supplier, amount, and due date — don't include line items."
Output is in the wrong format, wrong location, wrong structure, or wrong length
Outputs
Specify the output precisely. "Save as .xlsx in /outputs/ with one row per invoice and these exact column headers."

Two minutes with this table and you'll start diagnosing failures in your head automatically. The important thing is that the fix is always the same shape as the diagnosis — if work is off, you correct the work; if outputs are off, you correct the outputs. You don't rewrite the whole prompt.

A complete iteration, start to finish

Here's what good iteration actually looks like in a Claude Code session. One first attempt, one precise correction, done:

Session: Invoice-Processing
You: > Read every .pdf in /inputs/ (April invoices). Extract supplier, invoice number, amount, and due date. Save to /outputs/april-invoices.xlsx, one row per invoice. Claude: Done. Processed 14 invoices, saved to /outputs/april-invoices.xlsx. Three invoices didn't have invoice numbers — I skipped those. You: # noticing the "skipped three" note > Don't skip invoices without numbers — include them in the spreadsheet with "no invoice number" in that column. Claude: Updated. The spreadsheet now has 17 rows — the three previously skipped invoices are included with "no invoice number" in the invoice number column. You: > Perfect. Done.

Three messages. Good output. No rewriting of the prompt, no starting over, no frustration. The key moment is the one in the middle: "Don't skip invoices without numbers — include them with 'no invoice number' in that column." That single correction names what went wrong (skipping), says exactly what to do instead (include with a placeholder), and implicitly preserves everything else.

That's a whole lesson's worth of iteration in one message.

The undo button you didn't know you had

New to most learners

Rewind a conversation with checkpoints

Claude Code's VS Code extension quietly tracks every file change it makes. If an iteration made things worse, you can hover over any previous message and rewind — either undoing Claude's file changes, or forking the conversation from that point, or both.

You'll see three rewind options:

  • Fork the conversation — start a new branch from this message, keep the file changes as they are
  • Rewind the code — put the files back how they were at that point, keep the conversation history
  • Fork and rewind — both, i.e. a clean do-over from a specific point

This is genuinely new to most people — even power users rarely know it's there. It means you can try a correction, see if it makes things better or worse, and undo cleanly if not. No more "I wish I hadn't asked that" moments.

The four-correction rule

Sometimes iteration doesn't converge. You give a correction, it gets better but not right. You give another, it fixes one thing and breaks another. By the fourth round, you're fighting the session, not using it.

When that happens, stop. Rewriting a sharper prompt in a fresh session is almost always faster than the fifth iteration.

1-2 Iterate normally

Most outputs need one or two corrections to land. This is the sweet spot. You're adjusting small details — tone, structure, an edge case — and the result is getting closer each time.

3 Check your prompt

By the third round, ask yourself: is the original prompt actually clear? Iterating further often just papers over a fuzzy starting brief. Spend 30 seconds rewriting the prompt in your head before the next correction. If the mental rewrite is very different from what you typed, that's a sign.

4+ Start a fresh session

By the fourth round, end the session. Start a new conversation with a sharper, fuller IWO prompt that incorporates everything you've learned from the failed iterations. You're not losing work — you're getting out of a bad conversation faster. The fresh prompt usually lands on the first try.

Before you restart, capture what went wrong. Take 30 seconds to write down what the failing output kept getting wrong. That becomes the sharper instruction in your new prompt — or, if it's a recurring miss, a new line in your CLAUDE.md. Every frustrating session can make the next one better.

Feedback as a habit, not a repair

One more reframe worth sitting with. In most work, feedback comes up when something's wrong. That's the repair model — you only give feedback when you have to.

With Claude Code, feedback is the primary mode of working. First output, then correction, then better output. Not because Claude is worse than a human colleague, but because the loop is fast enough that refinement through feedback is actually the efficient path. A two-sentence correction gets you a better result in 20 seconds than a perfectly crafted five-minute prompt.

Once you internalise that, your prompts can be 60% right and you'll still ship faster than someone trying to write 100%-perfect prompts. The speed isn't in the first prompt. It's in the iteration.

The one thing to remember

Specific feedback fixes things. "Wrong" fixes nothing. Say exactly what to change.

Name what went wrong, name what to do instead, trust Claude to preserve what was already right. That's the whole pattern.

◆ Your action for this lesson

Practise bad-to-good iteration on purpose

In one of your project folders, deliberately give Claude Code a slightly vague prompt. Watch it do something not-quite-right. Then, instead of rewriting the prompt, practise giving a precise correction using the patterns from this lesson.

1. Send a deliberately rough prompt, e.g. "Sort out the files in /inputs/" 2. Look at the output and identify which part of IWO was off 3. Write a single correction that names what to change and what to keep 4. Send it. Notice how quickly Claude lands the right answer. 5. Bonus: try the rewind feature. Hover a message, pick a rewind option.

Deliberately inviting small failures is the fastest way to build the iteration reflex. Every time you do this, you're rehearsing the move you'll make on a real task under time pressure later.
