Claude Code: Setup, Workflow & Mastery Getting Claude Code onto your computer

Safety first — the four rules

Claude Code can delete files. Really delete them. Before you point it at anything that matters, you need four rules — and about ten minutes of backup setup. This is the least exciting lesson in the course, and the most important.

Everything in Module 1 until now has been a warm-up. You installed the tool, said hello, and saw it do something useful on a practice folder. Module 2 onwards, we start pointing Claude Code at real business files — invoices, spreadsheets, documents, client work. That's when things get properly useful, and also when things can go properly wrong.

This lesson is a checkpoint. Don't move past it until you've done the backup step at the end.

!
The common misconception

"Claude Code asks permission before it does anything, so I'm safe by default." True in normal mode, but not always. Permissions can be turned off. Folders can be opened on the wrong thing. Files can be overwritten, not just moved. The defaults are good — but relying on defaults alone is how people lose work.

The real risk, stated plainly

Claude Code is powerful because it acts on your files. That's the whole point. But that same capability means three things can go wrong:

  • Files can be moved to the wrong place and get lost in subfolders you'll never look in again.
  • Files can be overwritten when Claude creates a new version with the same name as an existing one.
  • Files can be deleted — occasionally on purpose because you asked, occasionally as a side effect of something else you asked for.

None of these are bugs. They're consequences of a tool that can act. A backup system makes every one of them recoverable. The four rules below keep you out of trouble in the first place.

Rule 1 — Always have a backup before using real files

Rule 1

Never run Claude Code on a folder whose files aren't backed up elsewhere

If the only copy of a file lives on your laptop, it's not safe. Not from Claude Code, not from coffee spills, not from stolen laptops, not from ransomware. A real backup means the file exists in two places at once, and one of them is off your machine.

The good news is that you almost certainly have a backup option available already — you may just not have turned it on.

Any one of these is fine. You don't need all of them. Pick the one you already have access to.

Time Machine (Mac)

Built in. Plug in an external drive, turn it on, done. Backs up continuously in the background.

File History (Windows)

Built in. Similar to Time Machine — external drive, on, done. Keeps versions too.

iCloud Drive / OneDrive

Keep working files inside the synced folder. Changes sync instantly and keep version history.

Google Drive / Dropbox

Same idea. Sync a folder. Work inside it. You get automatic version history for free.

A cloud-synced folder is not quite the same as a full backup. If Claude Code deletes a synced file, the deletion syncs too. But most sync services keep deleted-file history (usually 30 days) — so recovery is a few clicks in the web app. Good enough for our purposes, as long as you're not pausing the sync.

Rule 2 — Describe the outcome before you describe destruction

Rule 2

Write prompts that focus on the result, not on deletion

Claude Code does what you ask. So how you ask matters. Prompts that lead with verbs like "delete," "remove," "clean up," or "get rid of" are inviting the tool to be more aggressive than you probably want.

The fix is small but powerful: describe the end state you want, not the destructive action you imagine getting you there.

Here's the same intent, phrased two ways:

Two prompts, same goal, very different risk
✗ Risky: Delete all the duplicate files in this folder. ✓ Safer: Find duplicate files in this folder. For each pair, keep the one with the earlier date and move the rest to a subfolder called "_to_review". Don't delete anything.

Both prompts have the same goal — a folder with no duplicates. But the first invites Claude Code to decide which file is a duplicate, and act on that decision permanently. The second keeps deletion out of the conversation entirely and gives you a chance to review.

Treat "delete" as a word you rarely use. Move-to-archive. Flag-for-review. Rename-as-old. These are reversible. Deletion, much less so.

Rule 3 — Understand permission modes before you change them

Rule 3

Stay on normal or auto-accept mode. Bypass mode is off-limits for your real work.

Claude Code has several permission modes. They control how much Claude has to ask before it does things. You'll eventually want to change modes as your confidence grows — but you should know what each one does before you flip any switches.

Here are the four modes, honestly described:

Mode
What it does
Our take
Normal
Claude asks permission before every action that touches a file. This is the default.
Start here
Plan
Claude describes its full plan and waits for approval before doing anything. Great for big, multi-step workflows.
Safe
Auto-accept
Claude makes file edits without asking each time. Still refuses anything genuinely destructive without a check.

Bypass
Claude skips permission prompts entirely — formerly known as --dangerously-skip-permissions. Intended for sandboxed, offline environments only.
Off-limits

The simple rule for your real business work: stay on normal while you're learning the tool. Move to auto-accept only for workflows you've run several times and fully trust. Use plan mode when you're about to do something big that's going to touch a lot of files — like batch-processing a folder of contracts.

Bypass mode is not a productivity hack. It's an engineering feature designed for isolated test machines. Anthropic's own docs say to use it only in sandboxes with no internet access. If someone online tells you to run Claude Code in bypass mode to "skip all the annoying prompts," close the tab.

Rule 4 — Read before you approve

Rule 4

When Claude asks permission, read the actual request before clicking Accept

This sounds obvious. It isn't. After three or four accept clicks in a row, the fifth one starts feeling automatic. That's the one that'll surprise you.

When the permission popup appears in VS Code, it tells you exactly what Claude wants to do — which file, which action, which folder. Take the two seconds to look before clicking. If the request mentions a folder you didn't expect, or a file you don't recognise, reject it and ask Claude what's going on. Rejecting is always safe. You can always re-run the workflow.

The three requests most worth pausing on:

  • Anything that says "overwrite" or "replace an existing file." Moving is reversible; overwriting a file you didn't mean to is not.
  • Anything that operates on a folder outside the one you opened in VS Code. This shouldn't happen in normal usage — if it does, reject and investigate.
  • Anything that deletes a file or folder. Rare in most workflows. When it does happen, slow down.

If a workflow starts firing off prompts faster than you can read them, that's a sign the workflow is too broad. Stop it, write a sharper prompt, and try again.

If something goes wrong

It probably won't, especially with the four rules above in place. But if it does, stay calm. Most mistakes are recoverable, and the recovery path is usually the one you already have.

  • Moved to the wrong folder: ask Claude Code to move it back, or do it yourself in your file manager. No data lost.
  • Overwritten a file: check your backup system (Time Machine, File History, iCloud version history, Dropbox version history). Most keep at least the last 30 days.
  • Deleted a file: check the Trash / Recycle Bin first. Then your backup. Then your sync service's "recently deleted" folder.
  • Claude seems to have done something you can't explain: ask it. "Explain exactly what you changed in this folder in the last 10 minutes." It'll walk you through it. Then undo step by step.
The one thing to remember

A backup is a seatbelt. You hope you never need it. You never drive without it.

Rules 2, 3, and 4 keep you safe most of the time. Rule 1 — a working backup — is what saves you the one time the others don't.

The ten-minute checkpoint

Before you move to Module 2, spend ten minutes on the backup step. It's the only part of this course that's boring and the only part you can't skip.

If you already have a working backup system you know is running — Time Machine connected, File History active, iCloud Drive syncing, Dropbox mirroring — you're done. Move on. Tick the box, hit the quiz, and head to Module 2.

If you don't, or you're not sure, now's the time. Five minutes to set up, five minutes to let it take its first backup. Your future self will thank you the first time Claude Code surprises you.
