Claude Code: Setup, Workflow & Mastery The core workflow

Describing what you want — the new prompting

The prompts that worked brilliantly in ChatGPT will get you mediocre results in Claude Code. Not because your prompts are bad — because you're no longer asking a question. You're delegating a job. Here's the pattern that fits that shift.

By now, your project folder exists. Your CLAUDE.md is in place. Claude Code knows who you are and how you like things done. All that's left is describing today's task.

And this is where most people quietly hit a wall. They've used ChatGPT for years. They've got a feel for what makes a good prompt. They type one in — and Claude Code comes back with something vaguely right but not quite what they meant. The prompt isn't the problem. The shape of the prompt is.

!
The common misconception

"I'll just use the prompts I'm already good at." The chat Claude answers questions. Claude Code executes workflows. That's a different job, and it needs a different shape of prompt. Specifically: which files to read, what to do with them, and what to produce at the end. Treat every prompt like you'd brief a new assistant on a Monday-morning task, and the whole thing clicks.

The old shape vs. the new shape

Here's the same intent expressed two ways — once as a ChatGPT-style question, once as a Claude Code-style delegation:

In your browser (chat Claude)
> Can you summarise customer feedback themes for me?
In Claude Code
> Read the 40 review files in /inputs/. Group them by recurring theme, count how often each theme shows up, and write a one-page summary to /outputs/themes.md with headings per theme and two example quotes under each.

Read them side by side and the shift is obvious. The browser prompt is a question. The Claude Code prompt is a job specification. It names specific files to read, specific work to do, and specific outputs to produce.

This isn't because Claude Code is fussy. It's because Claude Code is acting. When the AI is about to touch your files — read forty of them, produce a new document — vagueness doesn't lead to a fuzzy answer. It leads to wrong work you have to undo.

↻
If you took AI Fundamentals You'll remember the ROLE / CONTEXT / TASK pattern from Lesson 3.1. The same principle still applies — but in Claude Code, the ROLE and most of the CONTEXT now live in your CLAUDE.md, already loaded at the start of every session. So the prompt itself focuses on the TASK — and breaks that task down more carefully, because the AI is about to act on real files.

The three parts of a Claude Code prompt

Every good Claude Code prompt names three things. Call them inputs, work, and outputs — because that's exactly what they are.

Part 1

Inputs

Which files or folders Claude should read. Be specific: the folder name, the file pattern, or the exact filename.

Read /inputs/*.pdf
Part 2

Work

What you want done with those files. The verbs and the rules — extract, categorise, summarise, group, rewrite.

Group by theme, count occurrences
Part 3

Outputs

What Claude should produce, where it goes, and what format. A markdown file, a spreadsheet, a set of renamed files.

Write to /outputs/themes.md

Miss any one of these, and you're inviting Claude to guess. Sometimes the guess is fine. Often it's "technically correct but not what you meant." Always it's a missed opportunity to have been precise in the first place.

Watch the three tiers in action

Let's see how the same intent improves across three tiers — vague, partial, and full inputs/work/outputs. Same folder, same underlying task: dealing with a pile of customer reviews.

Tier 1 — the vague prompt

Prompt Vague
> Sort out the customer feedback.

Claude Code will do something. It might categorise, it might summarise, it might pick a random file and rewrite it. You'll squint at the result, tell yourself it's fine, and move on. But you've spent no less effort than if you'd written a proper prompt — and the output is half as useful.

Tier 2 — partial specification

Prompt Partial
> Read the customer feedback in /inputs/ and summarise the main themes.

Better. You've named the inputs and roughly described the work. Claude will produce a reasonable summary. But you haven't said where to put the output, how long it should be, or what structure you want. So you'll get whatever Claude thinks is tidy — which might be a 2,000-word essay when you wanted a one-page brief.

Tier 3 — full inputs/work/outputs

Prompt Full I/W/O
Inputs: > Read every .txt file in /inputs/ (40 customer reviews from Q1 2026). Work: > Group reviews by recurring theme (e.g. "delivery speed", "product quality"). For each theme, count how many reviews mention it and pull two short representative quotes. Outputs: > Write the result to /outputs/q1-themes.md as a markdown document. One H2 heading per theme, ordered by frequency. Under each heading: the count, one sentence summarising the sentiment, and the two quotes as blockquotes.

That's the same task, fully briefed. You'll notice the prompt is longer — deliberately. Twenty extra seconds of typing removes twenty minutes of cleanup at the other end. Every time.

You don't need to literally write "Inputs:", "Work:", "Outputs:" as headings (though sometimes it helps). The three parts just need to be present somewhere in the prompt, in plain English.

Why this pattern works

Three reasons, all small but they compound.

It forces you to think before you type. You can't describe outputs vaguely if you're being specific — you have to know what "done" looks like. Half the value of this pattern isn't in the prompt; it's in the 30 seconds of thinking it makes you do first.

It gives Claude somewhere to put the answer. Without a specified output, Claude has to invent one — a file name, a location, a format. Its invention is fine, but it's rarely what you wanted. Specifying the output means the result lands exactly where you can find it.

It makes iteration precise. If a full IWO prompt goes wrong, you know exactly where it went wrong. "The output was the wrong format" points at the output spec. "It missed half the files" points at the input spec. Vague prompts fail vaguely; structured prompts fail in diagnosable ways.

When you don't need the full pattern

Not every Claude Code interaction needs to be a full inputs/work/outputs specification. Two small cases where you can relax.

Quick questions about the folder. "What's in this folder?" — "Summarise what you see in CLAUDE.md." — "Show me the contents of inputs/summary.csv." These are exploration, not delegation. A short prompt is correct here. Over-specifying is just friction.

Iteration on existing work. If Claude just produced an output and you want to tweak it — "Move the third quote up," "Use a table instead of bullets," "Shorten the headline" — you don't need a full IWO again. Claude still has the last result in context, so the correction is all the instruction it needs.

The full pattern is for new jobs — the moment you ask Claude to go and do something across files. Once the work is produced, you're back in a conversation and normal prompting rules apply.

A quick self-check before you press Enter

Before you send any Claude Code prompt that's going to touch your files, run this five-second mental check:

  1. Inputs — Have I named which files or folders Claude should read?
  2. Work — Is the task described with verbs specific enough that I'd know whether it was done right?
  3. Outputs — Have I said what should be produced, where it goes, and what format it should be in?

Three questions. Ten seconds. If any answer is "no" or "sort of," rewrite that part before you press Enter. This habit alone will save you more time than any other single thing in the course.

A useful shortcut. When Claude Code produces work that's almost right but not quite — "I wanted that as a CSV not markdown," "I asked for the top 10 not all of them" — look at which part of IWO you under-specified. That's the one to sharpen next time. Every miss teaches you something about your own prompting blind spot.
The one thing to remember

Inputs. Work. Outputs. If your prompt doesn't name all three, rewrite it.

The chat Claude answers questions. Claude Code executes workflows. Your prompt needs to brief a task, not ask about one.
