Skills — turning workflows into shortcuts

You've done the invoice workflow six times this quarter. You know the prompt by heart. Every time you type it out, you lose a little faith in the whole system. There's a fix — and it's one of Claude Code's most under-used features.

Welcome to Module 4. Modules 1 through 3 got you from nothing to doing real business work with Claude Code. Module 4 is about the stuff that separates casual users from people who've genuinely absorbed the tool — the moves that turn Claude Code from a thing you reach for into a system that quietly runs a chunk of your week.

We start with skills. If CLAUDE.md was Lesson 2.2's "single biggest leverage point," skills are Module 4's. They're how you turn the workflows you've been writing prompts for — invoice processing, CSV cleanup, LinkedIn drafts — into one-line shortcuts you invoke by typing /invoices or /clean-csv.

!
The common misconception

"Skills sound like a developer thing — custom commands, YAML, that whole scene." The surface looks technical, but the mental model is simple: a skill is a saved procedure with a short name. You type the name; Claude runs the procedure. The YAML header is two lines. Once you've done one, you'll be making them for every workflow you repeat.

Facts vs. procedures

The clearest way to understand what a skill is — and when to reach for one — is to compare it to CLAUDE.md, the thing you already know.

Module 2

CLAUDE.md — facts

Persistent context about who you are and how you work. Loaded automatically at the start of every session in that folder.

"I'm Priya, a brand strategist." "British English always." "Sign off with — Priya." Facts that apply everywhere

Module 4

Skills — procedures

Saved playbooks for specific repeated tasks. Loaded on demand when you invoke them, or when Claude decides they're relevant.

/process-invoices — extract these 7 fields, save to .xlsx /clean-csv — apply my 6-rule cleanup checklist /draft-posts — 5 LinkedIn posts, in my voice Procedures for specific jobs

A rough rule: facts go in CLAUDE.md. Procedures go in skills. If you find yourself writing a 15-line "how to do X" section in your CLAUDE.md, that's usually a skill waiting to be extracted. CLAUDE.md loads every session regardless of what you're doing, so packing procedures in there is wasteful. Skills only load when you actually use them.

How to know it's time for a skill

Here are the signals that a particular workflow has earned its way into a skill. Any one of these is usually enough.

Graduation signals

When to promote a workflow to a skill

  • You've run the same workflow 3+ times. Invoice processing, CSV cleanup, weekly reports. Three repeats means it's going to keep happening.
  • You find yourself scrolling back to find the prompt. If you're opening last month's Claude Code conversation to copy the prompt you used, that's a skill.
  • Your CLAUDE.md has grown a multi-step procedure. CLAUDE.md is for "always do this" facts, not "here's how to do X, step by step." Procedures moving to skills is the natural shape.
  • You'd hand the workflow to someone else if you could. Skills are exactly that delegation — except the "someone else" is Claude Code reading the skill at the start of your request.

What a skill actually is

A skill is a folder with one file in it called SKILL.md. The folder name becomes the shortcut you type. The file has a two-line header describing what the skill does, and then the instructions you want Claude to follow when you invoke it.

Here's the structure for a skill called process-invoices:

📁 ~/.claude/skills/ 📁 process-invoices/ 📄 SKILL.md 📁 clean-csv/ 📄 SKILL.md 📁 draft-posts/ 📄 SKILL.md

That's the whole thing. One folder per skill, one file inside each. The location matters — ~/.claude/skills/ makes skills available in every project on your machine. You can also put skills in a project-specific .claude/skills/ folder if they only apply to one client or context, but for a business owner starting out, personal skills at the user level are the right default.

A complete working skill

Let's turn the invoice workflow from Lesson 3.1 into a proper skill. Here's ~/.claude/skills/process-invoices/SKILL.md in full — you could genuinely copy this, change the field list, and have a working skill in five minutes:

📄 ~/.claude/skills/process-invoices/SKILL.md
--- name: process-invoices description: Extract structured data from a folder of invoice PDFs and produce a spreadsheet plus a summary report. Use when processing monthly or quarterly supplier invoices. --- # Process invoices For every .pdf file in /inputs/, extract: - Supplier name - Invoice number (use "—" if missing) - Invoice date (ISO format, YYYY-MM-DD) - Net amount (£, plain number) - VAT amount (£, plain number) - Gross amount (£, plain number) - Due date (ISO format) Write two outputs: 1. /outputs/invoices.xlsx — one row per invoice, columns in the order above. 2. /outputs/summary.md — a short report with: - Total invoices processed - Total gross spend - Top 5 suppliers by spend - Anything flagged for manual review ## Rules - Never invent values for missing fields — flag with "—" - If a date is ambiguous (e.g. DD/MM vs MM/DD), flag it - If a file can't be read (encrypted, corrupt), list it separately rather than skipping silently

That's it. Thirty-odd lines of plain English. No code. No configuration you need to understand. The two lines between the --- markers at the top are all the setup required — a name and a description. Everything else is just telling Claude what to do.

The description line is the important one. When Claude decides whether to invoke a skill automatically, it reads the description. "Extract structured data from a folder of invoice PDFs and produce a spreadsheet plus a summary report. Use when processing monthly or quarterly supplier invoices." — that wording is what lets Claude say "this sounds like what the user is asking for" when you later just write "sort out the April invoices." Write descriptions that include the words you'd actually use to ask for the task.

Using the skill

Once the file is saved, the skill is live. You don't need to restart anything. You invoke it one of two ways — and this is where the shortcut magic shows up.

Direct invocation — you type the command

Claude Code · Invoice-Processing project
> /process-invoices Reading SKILL.md… Reading CLAUDE.md… Scanning /inputs/ (found 47 PDFs)… Extracting invoice data… ✓ Processed 47 invoices ✓ Wrote /outputs/invoices.xlsx ✓ Wrote /outputs/summary.md Total gross spend: £284,320 across 18 suppliers. 2 invoices flagged for manual review.

One command. The full workflow runs. No need to describe the fields, the format, or the outputs each time — they're saved in the skill. You've just turned a 150-word prompt into a single slash-command.

Automatic invocation — Claude picks it up

Here's the part most people miss. Because Claude reads the skill's description at the start of every session, it can invoke skills on its own when your request matches. You don't even need to remember the slash-command:

Claude Code · Invoice-Processing project
> sort out this month's invoices Matching request to available skills… Invoking process-invoices skill. Processed 47 invoices, saved to /outputs/. Summary above.

Three words. Same result. This is why the description matters so much — a well-worded description means Claude picks up the right skill for the way you actually ask for things, not some artificial command-line incantation.

Where skills live (and who can see them)

Two locations matter for now. The third exists but isn't worth thinking about yet.

Personal skills — ~/.claude/skills/. Available in every project on your machine. This is where your workhorse skills live: /clean-csv, /draft-posts, anything you might want regardless of which folder you're in.

Project skills — .claude/skills/ in a project folder. Only available when that project is open. Use this for workflows that are specific to one client or one type of work — a /richardson-weekly-update skill that only makes sense inside the Richardson account folder, for example.

Plugin and organisational skills. These exist for teams and enterprise deployments. Not relevant for solo operators right now — mention it if you ever share skills across a team, but otherwise ignore.

A small but useful thing. Claude Code watches skill folders for changes — if you edit a SKILL.md during a session, the change takes effect on your next request. No restart needed. Which means you can iterate on a skill exactly the same way you iterate on a prompt: try it, tweak the SKILL.md file, try again.

What to watch out for

Writing the skill too tightly. The first version of /process-invoices works perfectly for the supplier PDFs you built it around, then fails on a slightly different format. Write skills a little looser than the first use case — "extract these fields if present" rather than "every invoice will have exactly these fields." Iterate when you hit the edge cases.

Writing the description too narrowly. If your description says "Process supplier invoices," Claude won't auto-invoke it when you say "sort out this month's receipts" — technically different words. Write descriptions in the language you'd actually use to ask for the task, with a few synonyms. "Invoices, receipts, bills, supplier statements" catches more natural phrasing than any one term.

Turning everything into a skill. One-off tasks stay as prompts. Skills are for work you'll repeat. Making a skill for something you'll do once is pure overhead — stick to the graduation signals from earlier.

Forgetting skills exist. This is the most common failure. You build four skills, use them for a week, then forget and start retyping full prompts. Worth a monthly ten-minute audit: open ~/.claude/skills/, remember what you have, consider what you've done more than three times and should promote.

The one thing to remember

Facts go in CLAUDE.md. Procedures go in skills. Three repeats is the promotion threshold.

A skill is a folder with one file. Two-line header, plain-English instructions, a slash-command you can invoke — or Claude picks up automatically. One hour of setup, saved forever.
