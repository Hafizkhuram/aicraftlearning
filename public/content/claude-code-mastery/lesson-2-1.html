Claude Code: Setup, Workflow & Mastery The core workflow

The project folder is everything

If you learn only one structural idea in this whole course, make it this one: the folder you open in VS Code is Claude Code's entire universe. Everything else in Module 2 builds on it.

Welcome to Module 2. Module 1 got you installed and oriented. Everything from here on is about how to actually work with Claude Code well — the mental models, the prompting patterns, and the small habits that separate the people who quietly get hours of their week back from the people who use Claude Code twice and forget about it.

It starts with one idea that sounds obvious and isn't.

!
The common misconception

"Claude Code can see anything on my computer." Nope. The only files Claude Code can read, write, or act on are the ones inside the folder you opened in VS Code. Not your Documents. Not your whole drive. Just that one folder and everything underneath it. This is both a safety feature and a design principle — and it shapes how you should organise your work.

The folder is the universe

When you open a folder in VS Code — File → Open Folder — that folder becomes the scope of everything Claude Code can do for you. The files inside are visible. The files outside don't exist as far as Claude is concerned.

Here's a concrete example. Say you've opened a folder called Invoice-Processing that contains this month's supplier invoices:

Visible to Claude

Inside the folder

  • Every file in Invoice-Processing/
  • Every subfolder and its contents
  • The CLAUDE.md file (Lesson 2.2)
  • Anything Claude writes during the session
Invisible to Claude

Outside the folder

  • Everything in your Documents
  • Everything on your Desktop
  • Your Downloads folder
  • Every other folder on your computer

If you ask Claude Code "can you look at my CV in Documents and update it based on this invoice?" — it can't. Not because it's refusing, but because Documents simply isn't in its world. You'd have to either drop the CV into the current folder, or close this session and open VS Code on a folder that contains both files.

This is why choosing the right folder to open is the single most important decision you make in every Claude Code session. Everything else — your prompt, your CLAUDE.md, your outputs — is scoped to that one choice.

Two kinds of folders you'll actually use

In practice, every folder you open in Claude Code falls into one of two shapes. Knowing which one you're dealing with sets the right expectations for everything else.

Project folder — durable
Scratch folder — ad-hoc

Most learners start with scratch folders and that's exactly right. The moment you find yourself doing the same task a second time, that's the signal to promote it into a proper project folder with a CLAUDE.md. We'll cover that in the next lesson.

How to structure a project folder

There's no one right way to organise a project folder, but there is a pattern that works well for 90% of business workflows. Here's what a healthy project folder looks like after a month or two of real use:

📁 Invoice-Processing/ 📄 CLAUDE.md # project instructions 📁 inputs/ # where you drop raw files invoice-2026-04-supplier-a.pdf invoice-2026-04-supplier-b.pdf invoice-2026-04-supplier-c.pdf 📁 outputs/ # where Claude writes results 2026-04-summary.xlsx 2026-04-report.md 📁 archive/ # processed files go here 📁 2026-03/ 📁 2026-02/ 📁 reference/ # things Claude should read but not change supplier-contacts.csv past-report-template.md

Four subfolders, each with a clear job:

  • inputs/ — the raw files you want Claude to process. You drop new work in here.
  • outputs/ — the files Claude produces. Never overwrite these manually; let the workflow refresh them.
  • archive/ — where processed inputs go after a run, usually dated by month or batch. Keeps the inputs folder clean.
  • reference/ — stable background files Claude should read to do the work well: templates, contact lists, style samples, glossaries.
This is a pattern, not a rule. Small projects might not need an archive/. Content projects might have a samples/ folder instead of reference/. Use the shape that fits — but notice how this pattern separates "what to work on" from "what you've produced" from "what to learn from." That separation is what makes the folder legible weeks later.

Watch what happens when you get it wrong

The clearest way to understand folder scoping is to see it fail. Here's the same exact question, asked in two different folders:

Scenario A · Opened in ~/Desktop (empty-ish)
> Summarise the April invoices in this folder. I don't see any invoice files in this folder. I can see a screenshot and two shortcut files, but nothing that looks like an invoice. Could you confirm the correct folder, or move the invoices here?
Scenario B · Opened in ~/Projects/Invoice-Processing/
> Summarise the April invoices in this folder. I found 14 invoices for April 2026 in /inputs. Here's the summary: • Total: £47,220 across 9 suppliers • Largest: Supplier B, £12,400 • Two invoices missing VAT numbers (flagged in report.md) Saved full breakdown to /outputs/2026-04-report.md.

Same prompt. Same AI. Completely different result. The only thing that changed was which folder was open when you asked.

When Claude Code replies "I don't see any files like that" or "I need more context," the answer is almost always that you opened the wrong folder — not that your prompt was bad. Check the VS Code sidebar. The folder name at the top of the sidebar is Claude's entire world right now.

Where to put your project folders

A small practical question that matters more than it seems: where on your computer should your project folders actually live?

My recommendation: a single Projects folder in your home directory, with each Claude Code project as a subfolder inside it. Something like:

📁 ~/Projects/ 📁 Invoice-Processing/ 📁 Meeting-Notes/ 📁 Client-Research/ 📁 Newsletter-Drafts/ 📁 _scratch/ # for one-off tasks

A few reasons this works:

  • Easy to find. One place to look when you want to open a session.
  • Easy to back up. One folder to include in Time Machine, Dropbox, or iCloud, and all your Claude Code work is covered.
  • Easy to tell what's what. The folder names describe real workflows, not "new folder 3."
  • Good separation from everything else. Your personal files, downloads, and random documents stay out of Claude's universe by default. Safer and less confusing.

If you live in cloud-synced folders — OneDrive, Dropbox, iCloud — put your Projects folder inside the synced area. You get automatic backups and cross-machine access for free.

The one thing to remember

The folder you open Claude Code in is its entire world. Choose it deliberately.

Good folders give Claude Code everything it needs to help you. Bad or ad-hoc folders lead to the "I need more context" replies that waste your time.

Common folder mistakes

Three mistakes show up constantly in the first few weeks of using Claude Code. Worth naming them now so you can skip the learning curve.

Opening your entire home folder "just to be safe." Tempting, but wrong. Claude Code becomes slower, less focused, and more likely to confuse your personal files with the work you actually want done. Worse, it massively widens the safety footprint — suddenly every file you own is in its universe. Always open the narrowest folder that contains what you need.

Opening the parent folder when you only meant to work in a subfolder. If you want to process invoices in ~/Projects/Invoice-Processing/inputs/, open ~/Projects/Invoice-Processing/ — not ~/Projects/. Otherwise Claude can see every other project too, and might cross-wire things.

Running everything in one giant "Claude-stuff" folder. Fine for your first week. A mess by week four. Multiple projects in one folder means Claude has to guess which workflow you're asking about every time, and the CLAUDE.md becomes impossible to write well. One folder per workflow is the rule.
