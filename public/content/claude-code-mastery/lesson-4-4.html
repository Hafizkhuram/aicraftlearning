Your 30-day plan — making it stick

Most people who finish a course like this close the browser and never open Claude Code again. This lesson exists so you're not one of them. What follows is four weeks of small, specific, daily moves — by the end, Claude Code is a real thing you use, not a thing you learned about once.

You've just finished fifteen lessons on how Claude Code works. Congratulations — you now know more about it than nearly anyone you'll meet. But knowing and using are different sports. The gap between them is the reason most professional courses don't change anything about the learner's week. This one can, but only if you walk out with a concrete plan for the next thirty days.

What follows is that plan. Four weeks, each with a specific focus, built around 15 minutes a day. That's genuinely all it takes — any more and real life stops you; any less and nothing compounds. Keep it small, keep it daily, and by the end of the month Claude Code is a tool you reach for naturally.

!
The common misconception

"I'll sit down for a proper 3-hour session next weekend and really dig in." Almost nobody actually does this, and those who try once rarely do it twice. The only reliable way to absorb a tool is a small amount, most days. Fifteen minutes repeated twenty-five times beats three hours repeated never.

The shape of the plan

The four weeks follow the arc of the course itself — each one applies a module you've already finished:

  • Week 1 — Foundations. Get Claude Code set up properly on the projects you care about. Applies Modules 1 and 2.
  • Week 2 — Your first real workflow. Pick one repeated task from your actual work and make Claude Code do it well. Applies Module 3.
  • Week 3 — Scale to three or four. Expand from one workflow to several. Compounding starts here. Still Module 3.
  • Week 4 — Skills and schedule. Promote your favourites into skills, schedule the ones that suit it. Applies Module 4.

You don't have to do this on a strict calendar — the "day" labels below are guidance, not rules. If work swallows Tuesday, roll it to Wednesday. The only thing that matters is that most days, you spend fifteen minutes on the current week's focus.

Week 1 — Foundations

Week 1

Get properly set up on work that matters

By Sunday Claude Code installed, 2-3 real project folders created, CLAUDE.md files written, first small task completed in each.
  1. Day 1 (Mon)

    Install if you haven't already. Claude Code through VS Code, Claude Pro subscription, confirm it runs on a test prompt. 15 minutes, mostly watching things download.

  2. Day 2 (Tue)

    Create your first real project folder. A client, a recurring admin task, a content project — something you actually care about. Add a /inputs/ folder and a /outputs/ folder.

  3. Day 3 (Wed)

    Write your first CLAUDE.md. Start with the template from Lesson 2.2. Who you are, how you write, what the project is for. 10-15 minutes is enough for version one.

  4. Day 4 (Thu)

    Run a real task. Not a test. Extract some data, clean a file, draft something small. The point is to prove the setup works on something that would have taken you actual time otherwise.

  5. Day 5 (Fri)

    Make a second project folder. Another chunk of your work — different client, different task type, different context. Repeat the CLAUDE.md + run-one-task pattern.

  6. Weekend

    Optional: a third project folder for a personal or side project. Skip if the week's been heavy.

Refresh from: Module 1 (Lessons 1.2-1.4) for setup, Lesson 2.1 for folder structure, Lesson 2.2 for CLAUDE.md.

Week 2 — Your first real workflow

Week 2

Pick one repeated task and nail it

By Sunday One workflow from Module 3 implemented end-to-end on real work, with a written prompt you can run again.
  1. Day 8 (Mon)

    Pick the workflow. Which of the four Module 3 shapes matches something you do regularly? Invoice extraction, CSV cleanup, research synthesis, or voice-matched drafting. Pick the one you'd use this week regardless.

  2. Day 9 (Tue)

    Prepare the inputs. Gather the real files (invoices, CSV, reports, writing samples) into the project's /inputs/ or /reference/ folder. 10 minutes of prep, the rest reviewing what you've got.

  3. Day 10 (Wed)

    Write the prompt carefully. Use the Module 3 patterns — specific question, clear outputs, flag-don't-guess, citations if research. Run it. Review what came back.

  4. Day 11 (Thu)

    Iterate once. Something will be wrong or slightly off. Use the checkpoint/rewind pattern from Lesson 2.4. Fix the prompt, not the output.

  5. Day 12 (Fri)

    Save the working prompt. Put it in a notes file, or better — you'll need it for Week 4. Do one final run to confirm it's reliable.

  6. Weekend

    Reflect. What took 45 minutes last week now takes 5. That's one workflow. Next week we multiply.

Refresh from: Module 3 generally. Specifically, the lesson matching your chosen workflow — 3.1 extraction, 3.2 cleanup, 3.3 research, 3.4 drafting.

Week 3 — Scale to three or four

Week 3

Expand the footprint

By Sunday Three to four working workflows across your real work. You stop reaching for manual approaches first.
  1. Day 15 (Mon)

    Look at last week's to-do list. Which items could have used a Claude Code workflow and didn't? That's your next candidate. Pick one.

  2. Day 16 (Tue)

    Run it through the three-question self-check (Lesson 4.3) before building. Specialist tool faster? "Close enough" causes harm? Your own thinking needed? If no to all three, proceed.

  3. Day 17 (Wed)

    Build workflow number 2. Same pattern as Week 2 — gather inputs, write careful prompt, iterate once, save the prompt.

  4. Day 18 (Thu)

    Pick a third candidate. From your to-do list, your inbox, your CRM, your content calendar. Ideally a different Module 3 shape from the first two.

  5. Day 19 (Fri)

    Build workflow number 3. Notice that it gets easier every time — the setup steps repeat, only the specifics change.

  6. Weekend

    Optional workflow number 4. Only if you're enjoying it. Don't force it — the point isn't volume, it's comfort.

Refresh from: Lesson 4.3 for the self-check, Module 3 for the workflow shapes.

Week 4 — Skills and schedule

Week 4

Turn workflows into systems

By Sunday Your best 2-3 workflows are skills. One of them is scheduled. You've graduated into a real system.
  1. Day 22 (Mon)

    Audit your workflows. Which of the 3-4 you've built have you actually run more than once? Those are the skill candidates — three repeats is the promotion threshold (Lesson 4.1).

  2. Day 23 (Tue)

    Build your first skill. Turn one of those saved prompts into ~/.claude/skills/<name>/SKILL.md. Two-line header, your instructions, save. Test with the slash-command.

  3. Day 24 (Wed)

    Build your second skill. Same pattern. You'll notice the description matters — write it in the language you'd use to ask for the task.

  4. Day 25 (Thu)

    Pick one for scheduling. Three-R test: reversible, reviewable, recurring. Invoice processing, weekly drafts, monthly reports — good. Anything that sends emails or posts — not yet.

  5. Day 26 (Fri)

    Schedule it. Use the natural-language pattern from Lesson 4.2. "Run /my-skill every Friday at 5pm." Confirm it's registered. Done.

  6. Weekend

    Wait. That's it. The schedule fires next Friday. By Monday of week 5, your first autonomous Claude Code output is sitting in your folder.

Refresh from: Lesson 4.1 for skills, Lesson 4.2 for scheduling, Lesson 4.3 for the three-R test.

The daily rhythm

What 15 minutes looks like

The small, repeatable shape

Each of the daily tasks above fits the same rough shape. Open the project folder (1 minute). Look at what the day calls for (2 minutes). Do the thing (10 minutes). Notice what worked and what didn't (2 minutes).

Do this before you start your "real" work, or at the natural pause after lunch, or in the last 15 minutes before you close your laptop. What matters is that it happens — not when.

When a day goes sideways

The bad-day protocol

Three rules for days you nearly quit

Some days you won't have 15 minutes. Some weeks will be awful. The difference between people who absorb a tool and people who don't isn't motivation — it's a light-touch recovery plan.

  1. Missed a day? Fine. Pick up where you left off, not where the calendar says you should be. The plan moves, not you.
  2. Missed three days? Drop to the minimum viable action — open Claude Code in one existing project, type anything, close. Momentum restored in 60 seconds.
  3. Missed a whole week? That week's focus becomes next week's focus. Don't try to "catch up" — it's not a race, it's a habit.

What you'll have by day 30

End state

A month from now, you'll have all of this.

  • ✓
    Two or three real project folders with CLAUDE.md files that actually reflect your work.
  • ✓
    Three or four working workflows that handle recurring tasks in 5 minutes instead of 45.
  • ✓
    Two or three skills you invoke with a slash-command or by describing the task naturally.
  • ✓
    One scheduled task that produces output without you thinking about it.
  • ✓
    The muscle memory of reaching for Claude Code when something's repetitive, structured, and low-stakes — and not reaching for it when it's the wrong tool.

That's not a student. That's a user. And a real working user of Claude Code is — genuinely — a meaningful advantage in most small businesses. This stuff still isn't common. You're not behind the curve; you're ahead of it.

The one thing to remember

Fifteen minutes a day for a month. Not three hours this weekend. Small, daily, forgiving.

By Day 30, you're not a course graduate. You're someone who uses Claude Code in their week. That's the only gap the plan is designed to close.
