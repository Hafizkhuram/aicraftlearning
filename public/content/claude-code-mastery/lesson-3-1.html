Claude Code: Setup, Workflow & Mastery Real business workflows

Processing documents in bulk

47 monthly sales CSVs. One executive dashboard. Three minutes of work instead of two hours. If you remember one thing from this module, make it this: Claude Code's sweet spot is doing one boring task to many files.

Welcome to Module 3. Modules 1 and 2 were about setting up the toolbelt — installing, learning how folders work, writing a CLAUDE.md, getting the prompting pattern into your muscle memory. The rest of the course is about doing real business work with that toolbelt. Not toy examples. Not developer scenarios. The boring, repetitive, hours-eating tasks that fill most knowledge workers' weeks.

This lesson kicks it off with the single most valuable workflow in the course: bulk document processing. Meaning: one repetitive task, applied to a folder of similar files, all at once.

!
The common misconception

"I'll use Claude Code for one complicated task." Backwards. Claude Code isn't at its best on the single complicated task — it's at its best on the simple, boring task you have to do fifty times in a row. Extract the amount from every invoice. Summarise every contract. Categorise every email. That's where hours disappear.

The worked example: 47 invoices into a spreadsheet

Let's walk through a full example start to finish. This is a real, plausible business task: you run a small consultancy, and at the end of each quarter your accountant asks you for a summary of every supplier invoice you paid. Normally that means opening 47 PDFs one by one, copying the numbers into a spreadsheet, and losing a morning to it.

Here's how that same job looks in Claude Code.

Step 1 — the folder

The project folder is already set up from Module 2. It looks like this:

📁 ~/Projects/Invoice-Processing/ 📄 CLAUDE.md # who I am, what this folder does 📁 inputs/ # where I drop this quarter's invoices supplier-a-001.pdf supplier-a-002.pdf supplier-b-april.pdf ... 44 more invoice PDFs 📁 outputs/ # where Claude writes the summary 📁 archive/ # dated folders for previous quarters

Step 2 — the CLAUDE.md

This file lives at the root of the folder and Claude reads it automatically at the start of every session. Short, specific, exactly the kind of thing Module 2 taught:

📄 CLAUDE.md
# Invoice-Processing project ## About this project This folder processes monthly and quarterly supplier invoices for a UK consultancy. All amounts are in GBP. Standard VAT rate is 20%. ## How the work flows - New invoices get dropped in /inputs/ as .pdf files - Claude extracts structured data and writes to /outputs/ - After each run, I manually move processed files to /archive/<quarter>/ ## Output conventions - Always use .xlsx for the main spreadsheet - Column order: supplier, invoice number, date, net, VAT, gross, due date - Dates in ISO format (YYYY-MM-DD) - If a field is genuinely missing, use "—" rather than inventing a value ## When in doubt Flag anomalies in a separate /outputs/flags.md rather than guessing.

Notice what's happening here. The CLAUDE.md tells Claude the rules of this particular workflow — currency, column order, date format, and crucially, what to do when data is missing. Write that once and you never explain it again.

Step 3 — the prompt

Now the instruction itself. Inputs, work, outputs — exactly the pattern from Lesson 2.3:

Claude Code · ~/Projects/Invoice-Processing
> Read every .pdf in /inputs/ (47 Q1 2026 supplier invoices). For each file, extract the supplier name, invoice number, date, net amount, VAT amount, gross amount, and due date. Save the combined result to /outputs/q1-2026-invoices.xlsx — one row per invoice, columns in the order from CLAUDE.md. Also write /outputs/q1-2026-summary.md with: - Total gross paid by supplier (ranked by size) - Any invoices with missing fields, listed separately - Count of invoices and total spend at the top

That prompt is slightly longer than a ChatGPT question would be, and deliberately so. It names every file to read, every field to extract, and both outputs it should produce — the spreadsheet and a summary markdown. Twenty seconds of typing saves the hour of "almost but not quite right" iterations that a vaguer prompt would trigger.

Step 4 — watching it work

Press Enter. Claude Code narrates as it goes. Here's a realistic trimmed view of what you'll see in the panel:

Claude Code · session
Reading CLAUDE.md… Found 47 .pdf files in /inputs/. Extracting from invoices… ✓ supplier-a-001.pdf ✓ supplier-a-002.pdf ✓ supplier-b-april.pdf ✓ supplier-b-may.pdf ⚠ supplier-c-003.pdf (due date unclear) ✓ supplier-d-invoice-44.pdf ... 41 more ✓ supplier-z-final.pdf Writing /outputs/q1-2026-invoices.xlsx (47 rows)… Writing /outputs/q1-2026-summary.md… Done. Processed 47 invoices in 2 min 14 sec. • 46 extracted cleanly • 1 flagged for review (supplier-c-003: due date unclear) • Total gross: £284,320 across 18 suppliers

That narration is important. On your first long-running job, the instinct is to worry it's stuck somewhere around file 12. It isn't. The running ticks are Claude's way of showing progress — once you've seen a batch run through, you'll stop staring at it.

A small habit worth forming. When a batch job is going to take more than a minute or two, start it and then go make a cup of tea. The work happens while you're away. Don't sit watching the ticks — that's just translating your time into the tool's time for no reason.

Step 5 — the output (and one iteration)

The q1-2026-summary.md file Claude wrote is sitting in /outputs/, ready to open. It looks something like:

📄 q1-2026-summary.md
# Q1 2026 Invoice Summary 47 invoices processed · 18 suppliers · £284,320 total gross spend ## Spend by supplier (top 10) 1. Supplier B — £52,400 (9 invoices) 2. Supplier F — £38,100 (6 invoices) 3. Supplier A — £29,200 (4 invoices) ... 7 more ## Flagged for review - supplier-c-003.pdf — due date unclear. Could be 31 March or 31 May depending on which digit is intended. Please confirm manually.

Is this perfect? Nearly. Say you notice one thing: the spreadsheet has the right data, but the date column is in US format (MM/DD/YYYY) instead of the ISO format your CLAUDE.md specified. Rather than rewriting the whole prompt, you give Claude one targeted correction — exactly like Lesson 2.4 showed:

Correction
> The dates in the spreadsheet are in US format. Convert them to ISO (YYYY-MM-DD) as the CLAUDE.md specifies, then re-save. Updated dates on all 47 rows. Saved.

15 seconds of iteration, everything else untouched. That's the full cycle: folder + CLAUDE.md + IWO prompt + targeted correction. Your accountant's quarterly summary is done.

Manual time vs. Claude time

Let's put actual numbers on it. Forty-seven PDFs, opening each one, copying the key fields into a spreadsheet, then producing a summary document:

Manual

~2 hours

3 minutes per PDF, plus the summary. A whole morning, if you're not interrupted.

Claude Code

3 minutes

Type the prompt, let it run, one targeted correction. Go do something else for most of it.

These numbers aren't theoretical. Real business owners report this kind of ratio routinely — two hours becoming three minutes, a Friday afternoon becoming a fifteen-minute task. The compounding effect, over a quarter or a year, is genuinely significant.

The pattern underneath the example

Invoices are a useful example but they're not the point. The point is the shape of the task. Any work that fits this shape is a Claude Code candidate:

The pattern

For each file in a folder, do the same thing, and produce a combined output.

Inputs: a folder of similar files. Work: one described task, applied to each file in turn. Outputs: either a single combined result, or one output per input file. That's it. Once you see this shape in your own work, you see it everywhere.

Here are four other places in a business week where that same shape shows up:

✉

Email triage

A folder of exported emails. For each one: categorise by intent (support, sales, spam). Output: a sorted list plus reply drafts for the real ones.

★

Review analysis

A folder of customer review text files. For each one: extract sentiment, themes, and the main ask. Output: a grouped summary with quotes.

§

Contract review

A folder of contract PDFs. For each one: summarise key clauses, flag non-standard terms. Output: one summary per contract, plus a master flag list.

↺

Meeting notes

A folder of raw meeting transcripts. For each one: extract decisions, action items, and owners. Output: a single structured summary per meeting.

Every one of those is the same workflow with different nouns. The prompt you just learned — inputs, specific work, structured outputs — adapts to each with about thirty seconds of rewriting.

What to watch out for

Three small things that trip people up on their first few bulk runs. Worth knowing before you hit them.

Mixed-type folders. If /inputs/ contains both PDFs and Word documents, Claude will read both unless you specify. Either keep the folder clean (one file type per folder), or name the pattern in your prompt — "every .pdf in /inputs/, not the .docx ones."

Scanned PDFs vs. text PDFs. If an invoice is a scanned image rather than a text PDF, Claude can still read it, but the extraction will be slower and occasionally less accurate on handwritten fields. If your accuracy drops unexpectedly on one file, check whether it's a scan — and expect to flag more fields manually on scans.

Very large batches. A hundred-file batch will work, but it's worth running your prompt on five files first as a sanity check. Fix any prompt issues on the small batch, then scale up to the full folder. Faster than discovering a prompt problem halfway through a hundred files.

The one thing to remember

One boring task, done to a hundred files, is Claude Code's sweet spot. If you're doing it manually, stop.

The pattern is "for each file in a folder, do X, produce Y." Once you see it in your own work, you'll start finding it everywhere.
