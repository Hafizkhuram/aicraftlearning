Claude Code: Setup, Workflow & Mastery The core workflow

Meet CLAUDE.md — your project's memory

Every new Claude Code conversation starts with amnesia. It doesn't remember who you are, what the project is, or how you like things done. This one file fixes that — and it's the single biggest leverage point in the whole course.

If Lesson 2.1 was about where Claude Code works (the folder), this lesson is about what it knows when it gets there. Because by default, the answer is: nothing. Every session, you're introducing yourself from scratch.

That's tedious. It's also unnecessary. Claude Code automatically reads one specific file at the start of every session — a plain markdown file called CLAUDE.md — and treats it as persistent context for the whole project. Write it well once, benefit from it forever.

!
The common misconception

"I just need to write better prompts." Not really. If you find yourself typing the same explanation into Claude Code every Monday — who you are, what the project is, how to format the output — that's not a prompting problem. That's a missing CLAUDE.md. The fix isn't a cleverer prompt; it's writing down the context once and letting Claude read it automatically.

What a CLAUDE.md actually is

A CLAUDE.md is a plain text file — just markdown, like the notes you might already write. You put it in your project folder, and Claude Code reads it at the start of every session in that folder. No setup, no special command, no settings to configure. The filename is the magic.

Inside it, you write down the stuff you'd otherwise have to re-explain: who you are, what the project is for, what style you prefer, what Claude should and shouldn't do. That's the whole idea.

Think of it like the briefing document you'd give a new hire on their first Monday — except you only have to write it once, and it never gets forgotten.

Watch what happens without one

Here's a small but revealing example. Same folder, same task — drafting a client update email — run twice. First time without a CLAUDE.md:

Session 1 · No CLAUDE.md · Folder: Client-Updates/
> Draft a project update email for the Richardson account, based on the notes in /inputs/richardson-week-14.md. Dear Richardson Team, I hope this email finds you well. I wanted to provide you with an update on the progress of your project this week. We have successfully completed the deliverables outlined for this sprint, including the initial research phase and the drafting of the brand positioning document. We will continue to work diligently to ensure the successful delivery of the project, and please don't hesitate to reach out with any questions or concerns. Best regards, [Your name]

Technically correct. Competently written. And painfully generic. "I hope this email finds you well" is the telltale sign of an email written by an AI that has no idea who's speaking or to whom.

Now, exactly the same task — but this time there's a CLAUDE.md in the folder explaining who you are and how you write:

Session 2 · With CLAUDE.md · Folder: Client-Updates/
> Draft a project update email for the Richardson account, based on the notes in /inputs/richardson-week-14.md. Subject: Week 14 — brand positioning doc is with you Sarah, James, Quick one. The first draft of the positioning doc landed in your shared drive this morning — look for "RICH-brand-v1". Three things worth flagging: • We've gone harder on the "quiet confidence" angle you mentioned in the kick-off. I think you'll like it. • The messaging pillars need a call before we lock them. Thursday afternoon good? • We're on track for delivery by the 28th. No action needed from you today — just wanted you across it. Shout with any thoughts, Priya

That's a different email. Specific names, specific files, your voice, your cadence, your sign-off. The underlying AI is the same. The prompt is the same. The only thing that changed is that Claude now has a file in the folder telling it who you are and how you write.

Multiply this across every email, report, summary, and document you ever ask Claude Code to produce in that folder. That's what a good CLAUDE.md buys you.

What to put in a CLAUDE.md

Writing a CLAUDE.md sounds like it should be hard. It isn't. A good one has four sections, and you can draft all of them in fifteen minutes.

About me (who's speaking)

Your name, your role, your business, your audience. Two or three sentences is plenty. Claude uses this to choose tone, voice, and the right professional register.

About this project (what this folder is for)

What kind of work happens here, who the outputs are for, what success looks like. If this is an invoices folder, say so. If it's a client-writing folder, name the clients.

How I like things done (style and conventions)

British English. Sentence case headings. Short paragraphs. Bullet points over prose for action items. Always sign off with "—Priya". Whatever matters for this project — write it down.

What to avoid (the red lines)

The things Claude gets wrong often enough that you want to name them. No AI-speak ("I hope this email finds you well"). No em-dashes. Don't use "leverage" as a verb. Whatever your own pet peeves are.

That's it. Four sections. No technical setup. No special syntax. Just plain English written to a helpful assistant.

A full working template

Here's a complete CLAUDE.md you can copy straight into one of the project folders you created in the last lesson, then adapt to fit your own work. This is the one I'd use for a client-writing project:

📄 CLAUDE.md
# Client-Updates project ## About me I'm Priya, a freelance brand strategist. I write for small and medium business clients — mostly UK-based, professional services and B2B. My clients hire me for strategic clarity, not just pretty copy. ## About this project This folder is for weekly and monthly client updates. Each client has a subfolder in /clients with their voice samples and brand notes. I work with three active clients at the moment: Richardson Consulting, Olive & Co, and Haven Design. ## How I write - British English always — colour, organise, realise - Short paragraphs. Two or three sentences is often plenty - Sentence case headings, never title case - Direct and warm. I use the client's first name in emails - I sign off "— Priya" with an em-dash and my name only - I use "we" when talking about delivery, "I" when sharing views ## Avoid - "I hope this email finds you well" — never, under any circumstance - Corporate filler: "going forward", "leverage", "synergies", "circle back", "at the end of the day" - Exclamation marks for emphasis - Overly long sign-offs ("Best regards, and please don't hesitate to reach out…") - Bullet points in every email — use prose where it flows ## When in doubt Before drafting anything new for a client, read their folder under /clients/<client-name>/ for voice samples. Match the cadence and tone you see there — not what you assume.

A few things worth noticing about the template above.

It's specific. Not "use British English when appropriate" but "British English always — colour, organise, realise." Specific instructions get followed; vague ones get ignored. That's a consistent pattern worth remembering every time you write one of these.

It's short. About 30 lines. Anthropic recommends keeping CLAUDE.md files under 200 lines — and honestly, you rarely need even 100. Long files lose focus. Short, specific files win.

It ends with a "when in doubt" clause. This is the most underused pattern. Tell Claude what to do when it's uncertain — usually, read something specific in the folder, or ask you. That one line removes hundreds of future misunderstandings.

Two ways to create yours

You have two routes. Pick whichever feels easier.

Option A — Ask Claude Code to generate one

Open your project folder in VS Code. Type /init into Claude Code's prompt box. Claude will look at the folder, make intelligent guesses about what the project is, and draft a starting CLAUDE.md for you. It'll ask a few questions along the way.

You're not done — the draft is a starting point, not a finished file. But it gets you past the blank page, which is the hardest part.

Option B — Write it yourself using the template above

Right-click inside your project folder in VS Code's sidebar. Choose New File. Name it exactly CLAUDE.md (capital letters matter on some systems). Paste the template, then edit each section to match your own life and work.

This takes 10-15 minutes done properly. Worth every second.

A small tip. Whenever you find yourself typing a correction into Claude Code — "no, use bullet points not paragraphs," "call them clients not customers" — that's a signal to add the rule to your CLAUDE.md. Every correction you type twice belongs in the file instead. It's how the file grows into something that genuinely knows your business.

Checking Claude is actually reading it

A useful sanity check, especially the first time: make sure Claude Code is actually loading your file. Type /memory into the prompt box. Claude will show you every CLAUDE.md currently loaded into the session. If yours is listed, you're good. If it isn't, check two things:

  • The filename is exactly CLAUDE.md — capital C-L-A-U-D-E, capital M-D. Not claude.md, not CLAUDE.MD.
  • The file is at the root of the folder you opened in VS Code, not buried in a subfolder.

If both those are right and it still isn't loading, try closing the session and reopening the folder — Claude reads the file at session start, so brand-new files sometimes need a refresh.

The one thing to remember

A great CLAUDE.md is the difference between a stranger and a colleague. Write one for every project you'll come back to.

One file, written in plain English, read automatically at the start of every session. It's the most valuable fifteen minutes you'll spend in this course.

A note on "auto memory"

You may see the words "Writing memory" appear in Claude Code's interface from time to time. That's a separate, automatic feature called auto memory — Claude writing its own notes about things it's learned during your sessions (conventions you've corrected, preferences you've mentioned).

You don't need to worry about this for now. It's on by default, it's stored locally on your machine, and it complements CLAUDE.md rather than replacing it. You write CLAUDE.md to shape Claude's behaviour deliberately. Claude writes auto memory to remember the small things between sessions without you having to nudge it.

If you're curious, type /memory and browse the folders you see there. If you want to turn auto memory off, there's a toggle in the same place. For most learners on this course, leaving it on is the right call — it quietly makes Claude a bit more useful with each session.

Write a CLAUDE.md for one of your project folders

Pick one of the three project folders you created in the last lesson. Write a CLAUDE.md for it — using /init, the template above, or a mix of both. Spend ten to fifteen minutes. Keep it short, specific, and honest.

1. Open one of your project folders in VS Code 2. Either type /init or create a new file called CLAUDE.md 3. Fill in: About me / About this project / How I write / Avoid 4. Save the file, then run /memory to confirm it's loaded 5. Ask Claude Code a small task. Notice the voice match.

This is the highest-leverage single action in the whole course. The next time you open that project, Claude will already know who you are. That's not nothing.
