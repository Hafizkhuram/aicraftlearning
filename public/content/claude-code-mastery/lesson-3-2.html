Turning messy data into clean reports

A client sends you a CSV that would make a developer cry. Missing fields, mixed date formats, inconsistent capitalisation, duplicates, the works. Here's how Claude Code turns two hours of cleanup into five minutes — and then writes the report on top.

Every business owner has seen this file. It lands in your inbox from a client or a tool export, you open it, and your heart sinks a little. Rows with empty fields. Dates in three formats. "LONDON" in one row and "london" in the next. One or two entries duplicated. And you're the one who has to make sense of it before the meeting on Thursday.

This is the second workflow where Claude Code pays for itself immediately. Spreadsheets and CSVs are its sweet spot — not just because it can read them, but because it can fix many different kinds of mess in a single pass, and then generate the report you actually wanted on the other side.

!
The common misconception

"I should clean the data myself and use Claude for the analysis." The opposite — the cleanup is the tedious part humans are worst at. Claude Code can apply five or ten cleanup rules at once, consistently, without getting bored. Describe the rules once, run them on any messy file, repeat every time a client sends you a new export.

Two patterns, one module

Before the worked example, a framing point. Lesson 3.1 taught you one shape of bulk work — the same task applied to many files. This lesson teaches a different shape. It's worth seeing them side by side:

Lesson 3.1

Many files, one transformation

"For each file in this folder, extract the same fields and combine into one output." Invoices, contracts, reviews.

Lesson 3.2

One file, many transformations

"Take this one messy spreadsheet and apply five or ten cleanup rules at once, then produce a clean version plus a report." CSVs, exports, data pulls.

Most of your Claude Code work will be one of these two shapes. The prompting pattern adapts — but the diagnostic habit from Module 2 still works: inputs, work, outputs.

The messy file

Let's make this concrete. A client has sent you their customer list as a CSV — 347 rows, supposedly all active clients. You open it and find the usual horror show. Here's a representative slice:

📊 customers-2026-raw.csv Messy
NameEmailCityJoinedStatusSpend
Sarah Patel sarah@example.co.uk LONDON 15/3/2024 Active £4,200.00
James Okafor james.o@example.com Manchester 2024-03-22 active 2850
Priya Shah priya@example.com london Apr 5, 2024 Active 6100
Sarah Patel sarah@example.co.uk LONDON 15/3/2024 Active £4,200.00
Oliver Bennett — Bristol 11.04.2024 ACTIVE 1950
Marta Kowalski marta.k@example.com Edinburgh 2024-04-18 Inactive —

Squint at that and you'll count the problems without even trying:

What's wrong with this data
  1. Inconsistent capitalisation — "LONDON", "london", "Manchester" all mean the same thing but won't group together.
  2. Mixed date formats — DD/MM/YYYY, YYYY-MM-DD, "Apr 5, 2024", and DD.MM.YYYY, sometimes with extra whitespace.
  3. Inconsistent status casing — "Active", "active", "ACTIVE", "Inactive".
  4. Mixed currency formatting — "£4,200.00" vs. raw "2850". Same kind of value, different conventions.
  5. Duplicate rows — Sarah Patel appears twice, identical.
  6. Missing values — Oliver Bennett has no email, Marta has no spend. Both need handling explicitly.

Done manually, this is an hour of Find-and-Replace, date parsing, and carefully hunting duplicates in Excel. Done with Claude Code, it's a single prompt.

The cleaning-checklist pattern

Here's the prompting move that turns CSV cleanup from an art into a repeatable skill. Instead of asking Claude to "clean up this file" (vague, invites guessing), you give it an explicit checklist of cleanup rules.

This is the same idea as inputs/work/outputs from Lesson 2.3, but with the "work" part broken into a list. Claude applies each rule in order and reports back on what it changed. Every rule gets handled consistently, across every row.

The template

A starter cleaning checklist

For almost any messy CSV, the same six cleanup rules handle 90% of real-world problems:

  • Whitespace — trim leading and trailing spaces from every cell
  • Capitalisation — title case for names, sentence or lower case for categorical fields like cities, all-lower for emails
  • Dates — convert to a single format (ISO YYYY-MM-DD is the safest)
  • Currency and numbers — strip symbols, use plain numbers, note the currency in the column header or a separate column
  • Duplicates — remove rows where all fields (or the key identifier) match
  • Missing values — decide in advance: flag with a placeholder, exclude from analysis, or leave blank

You don't need to reinvent this list every time. Write it into your project's CLAUDE.md once, and every future CSV cleanup inherits the same conventions.

The full prompt

Here's how you'd actually brief the work. Notice the shape: inputs (the file), work (the checklist), outputs (two files — one clean, one report):

Claude Code · Client-Data project
> Read /inputs/customers-2026-raw.csv (347 rows). Apply these cleanup rules: 1. Trim all leading/trailing whitespace 2. Capitalise cities as title case (London, Manchester) 3. Standardise Status to title case (Active, Inactive) 4. Convert all dates to ISO format (YYYY-MM-DD) 5. Strip currency symbols from Spend, keep as plain numbers 6. Remove exact duplicate rows 7. Flag any missing values with "—" and list them separately Save the cleaned file to /outputs/customers-2026-clean.csv. Then write /outputs/cleanup-report.md with: - Original row count vs. clean row count - What was changed (counts per rule) - Which rows had missing values and what's missing - Top 5 cities by client count - Active vs. inactive breakdown with total spend per group

That prompt is about 150 words. It tells Claude Code exactly what mess to handle, which decisions to make, and what two outputs to produce — a clean data file for further work, and a human-readable report for you.

Turn the checklist into a reusable workflow

Here's where compounding kicks in. Your client sends this same kind of export every month. Instead of re-writing the cleanup prompt every time, bake the rules into your project's CLAUDE.md:

📄 CLAUDE.md
# Client-Data project ## Standard CSV cleanup rules Apply these to every customer CSV I drop into /inputs/: - Trim whitespace in every cell - Cities: title case - Status values: title case - All dates: ISO format (YYYY-MM-DD) - Currency: strip symbols, keep plain numbers. Assume GBP. - Remove exact duplicate rows - Flag missing values with "—" rather than guessing ## Standard report For every cleanup run, also produce: - Count summary (in vs. out) - Top cities by count - Status breakdown with spend totals ## When in doubt List edge cases in /outputs/flags.md rather than silently normalising them.

Now future cleanup jobs need just a short prompt: "Clean the file in /inputs/ using the standard rules. Produce the standard report." Everything else — the checklist, the format conventions, the report structure — is already loaded.

This is the compounding effect Module 2 promised. Every workflow you run through Claude Code leaves behind knowledge you can reuse. The first cleanup takes five minutes of prompt-writing. The twelfth one takes ten seconds. That's the difference between using a tool and building a system.

What to watch out for

A few practical notes from doing this often.

Always keep the original file. Claude Code writes to /outputs/, not back over your original. That's by design — if you ever need to re-run with different rules, you want the raw version intact. Make sure your CLAUDE.md enforces this.

Sanity-check the row count. If the cleaned file has dramatically fewer rows than the original, something went wrong — Claude probably over-aggressively dedupe'd, or dropped rows that should have been flagged. Check the cleanup report's count summary as the first thing you read.

Don't let Claude guess at ambiguous data. If a date could be DD/MM or MM/DD and Claude isn't sure, you want that flagged, not silently resolved. This is why the "when in doubt, list in flags.md" rule belongs in every CLAUDE.md for this kind of work.

Bigger files need small-batch testing. Just like Lesson 3.1, if the CSV has 10,000+ rows, test your checklist on a slice of 50 first. Fix any wobbles, then scale up.

The one thing to remember

Messy CSV in, clean report out. This is the fastest ROI task in the whole course.

A cleaning checklist plus a CLAUDE.md plus one prompt — and an hour of tedious data work becomes five minutes. Every month.
