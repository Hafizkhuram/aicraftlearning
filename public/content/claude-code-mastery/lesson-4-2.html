Scheduling Claude Code — running on autopilot

You walk to your desk on Monday morning. The weekly report is already on your screen. Claude Code wrote it at 7am while you were making coffee. This is the lesson where you stop running workflows yourself and let time run them for you.

Lesson 4.1 turned your repeated workflows into skills — one-line shortcuts you invoke with /process-invoices. This lesson goes one step further: invoking the skill yourself is still a thing you have to remember to do. Scheduling removes that last piece. The skill fires itself at 9am every Monday, and the output is waiting when you sit down.

Used well, this is where Claude Code stops being "a tool you use" and starts being "a small assistant that just gets on with it." Used badly, it's a great way to accumulate tasks that silently stopped firing three weeks ago. Let's make sure you end up in the first category.

!
The common misconception

"Scheduling means Claude Code runs by itself, fully hands-off, on any schedule I want." Sort of. There are three different ways to schedule Claude Code, each with different trade-offs — and choosing the wrong one is the most common reason scheduled tasks quietly fail. The difference between "my machine has to be on" and "runs in the cloud regardless" matters a lot for what you'll actually trust to your automations.

The three ways to schedule

Claude Code offers three separate mechanisms for running prompts on a schedule. They look similar on the surface but behave very differently. Getting the right one for the job is most of the battle.

/loop — session scheduling

Easiest

A built-in command that re-runs a prompt every few minutes while your Claude Code session is open. Type /loop 5m check the deploy and it repeats that check every 5 minutes until you close the session.

Best for Short-term polling. Watching a long-running build, monitoring a deployment, waiting on a PR. Not real automation — the loop stops the moment you close Claude Code.

Desktop scheduled tasks

Most useful for you

Tasks that live on your machine and fire on a schedule even when Claude Code isn't open. Your computer has to be on (and awake), but the task can read your local files and run any workflow you've built.

Best for The Monday-morning report. Weekly invoice processing. Daily briefing. Any recurring task you want to land on your desk at a specific time, using files already on your machine.

Cloud routines

Most robust

Tasks that run on Anthropic's infrastructure without needing your machine on. They don't have access to your local files — they fetch what they need from the internet or connected services. Set them up once and they run forever.

Best for Unattended work. Scraping web sources, responding to emails, tasks that need to run at 6am even if your laptop is off. More setup, more robustness.

For most non-technical business owners, Desktop scheduled tasks are the right default. They're the sweet spot — proper automation that fires without a Claude Code session, full access to your files, and your computer is almost certainly on during working hours anyway.

Skills + schedules = actual automation

The connection from Lesson 4.1

This is the lesson where skills really pay off.

Schedules and skills are a natural pair. On their own, a schedule has to carry the whole prompt — all the instructions, all the conventions, all the output rules. That gets unwieldy fast. But if you've already captured the workflow as a skill, the scheduled prompt becomes a single line: /process-invoices. Everything else is in the skill. Simple to schedule, easy to update later by editing the skill rather than the schedule.

So the workflow is: build the skill first (Lesson 4.1), then schedule the skill (this lesson). You're not writing a big prompt on a timer — you're triggering a named procedure you've already tested.

Setting up a real schedule

Let's make this concrete. Say you want your weekly invoice processing to fire every Friday at 5pm — so when you open your laptop on Monday morning, the previous week's invoices are already processed and the summary is sitting in your outputs folder.

The mechanics of Desktop scheduled tasks are set up through Claude Code's scheduling tools. You don't need to write cron expressions by hand — you describe what you want in natural language, and Claude sets it up:

Claude Code · Invoice-Processing project
> Schedule /process-invoices to run every Friday at 5pm. Creating scheduled task… ✓ Scheduled: "process-invoices" will run weekly at 17:00 every Friday (local time). Task ID: a3f91b2c > what scheduled tasks do I have? You have 1 scheduled task: • a3f91b2c — /process-invoices Every Friday at 17:00

Three things are happening under the hood, none of which you need to think about:

  • Claude converts your plain-English schedule into a proper time specification (the kind of thing developers call a "cron expression" — 0 17 * * 5 in this case)
  • The task is saved on your machine, ready to fire at the scheduled time
  • Every Friday at 5pm, Claude Code wakes up, runs /process-invoices, and produces the output files in your project folder

You can list, modify, or cancel tasks the same way — just describe what you want. "Cancel the Friday invoice task" works. So does "change the invoice schedule to Thursday at 4pm instead."

Times are local, not UTC. When you say "Friday at 5pm," Claude Code schedules it for 5pm in your local timezone. If you travel, the tasks travel with you — 5pm in London when you're in London, 5pm in Lisbon when you're in Lisbon. Helpful for working life, less helpful if you wanted the task to always fire at a global UTC time (rare case, but worth knowing).

Useful schedules for real work

The examples below are the schedules that earn their keep for most business users. Every one is a scheduled skill you've already seen in Module 3 and 4.1:

Mon 7:00 am

Weekly briefing

Run /research-sources on a folder of client emails and meeting notes. Land a weekly-briefing.md on your desk before work starts.

Fri 5:00 pm

Invoice processing

Weekly /process-invoices on everything dropped into /inputs/ that week. The accountant gets Monday what was booked Friday.

Mon 8:00 am

Content drafts

Fire /draft-posts against the week's themes (stored in a briefs.md file). Wake up to five draft posts ready to edit.

Last of month

Client reporting

Monthly /clean-csv plus a summary skill. First of the month, every month, every client's report is already there.

The pattern in all four: a named skill that already works, fired on a time that matches your natural work rhythm. Not a science. Just "the thing I meant to do Friday" being done by Friday without me.

The rules of good scheduling

Three habits separate people whose scheduled tasks keep working from people who eventually abandon them.

Schedule things that produce visible output. If a scheduled task fails silently, you might not notice for weeks. Every good scheduled workflow writes something to a folder you actually open — a markdown summary, a spreadsheet, a report. The output file's timestamp tells you whether the last run actually happened.

Match the schedule to your rhythm, not the computer's. Tasks fire at times you'll naturally check the output. Friday 5pm is good for Monday-morning review. Sunday 11pm is terrible — you won't see the output for hours. Pick moments that set up something you'll do next.

Do a monthly "are my schedules still running?" check. Once a month, ask Claude Code: "What scheduled tasks do I have?" Check the output files have fresh timestamps. If something's stopped working (maybe the project folder moved, maybe a skill got renamed), catch it fast.

Scheduling has real limits. Be honest about them. Your machine has to be on for Desktop tasks to fire. They don't catch up missed runs — a task scheduled for 9am that missed because your laptop was closed at 9am just fires the next time it comes due. For anything genuinely mission-critical, you want Cloud routines, a reminder system, or a human check. Scheduled tasks are a productivity win, not a reliability guarantee.

When NOT to schedule

Scheduling has a cost — every scheduled task is a thing you're trusting to run without supervision. Some things deserve that trust. Some don't.

Don't schedule anything irreversible. Sending emails, making payments, posting to social media, deleting files. If the task is going to do something you can't undo, it should require you clicking "go." Scheduling such things quietly goes wrong eventually, and you find out because a client emails you back.

Don't schedule tasks whose output you'll never check. A weekly report that lands in a folder you don't open isn't automation, it's digital compost. Only schedule things where the output meaningfully lands in your working life.

Don't schedule one-offs. "Remind me Thursday at 3pm to email Sarah" — that's a calendar reminder, not a scheduled task. Claude Code can do one-off reminders, but for something you'll do once, just use your actual calendar.

Good scheduling candidates are recurring, reviewable, and reversible. Three Rs to hold in your head.

The one thing to remember

Schedule skills, not prompts. Pick times that set up something you'll do next. Reviewable, reversible, recurring.

Desktop scheduled tasks are the right default for most business users. Build the skill first, then schedule it — and check your schedules every month.
