Building AI Agents & Workflows Build your first agent

What makes an agent different

A workflow runs a recipe. An agent is the cook who decides what to make. One of those is much harder to keep in line — and much more valuable when the question is "what should we make?"

Welcome to Module 3. You're about to build something genuinely different from Build #1. Before we do, this short lesson reframes what "agent" actually means — because the word gets used loosely, and the mental model you walk in with shapes every decision you'll make in the next three lessons.

!
The common misconception

"An agent is just a fancier workflow — same thing with AI sprinkled on top." It isn't. Adding a Claude node to a workflow doesn't make it an agent. An agent is a different architecture: one where the path through the work isn't fixed in advance. The agent reads the task, decides what to do next, does it, looks at the result, and decides again — in a loop — until the task is done. That loop is the agent. Remove it and you're back to a workflow, no matter how clever the prompt.

The three ingredients

An agent is the combination of three things. Miss any one and what you've built is something else — possibly useful, but not an agent.

01

Tools things the agent can call

Tools are functions the agent can invoke on demand — search the web, read a URL, query a database, send a message, append to a spreadsheet. The agent doesn't know ahead of time which tools it will use; it picks them as it goes, based on what the task needs.

In practice For Build #2 (the research agent), the tools will be a web search, a URL reader, and a Google Sheets writer. The agent decides which to call, in what order, for each competitor.
02

Memory what the agent remembers

An agent keeps track of what it's seen and done so far. Without memory, each tool call is blind to every previous one — you'd get the same Google search repeated endlessly. Short-term memory lives inside the current session; optional long-term memory carries facts across sessions.

In practice Session memory lets the research agent notice "I've already checked the homepage — now try the blog." Cross-session memory could let it compare this week's findings to last week's. We'll use session memory only in Build #2 — it's enough.
03

The decision loop the shape that makes it an agent

The agent receives an input, reasons about what to do, calls a tool, observes the result, and decides the next step — or stops, if the task is done. That loop is the difference. A workflow goes A → B → C, always. An agent goes "what's needed? → call tool → look at result → repeat" until it judges the work complete.

In practice You'll see this loop explicitly in Build #2's execution log — each "turn" is a step in the reasoning, and you'll be able to read the agent's thinking line by line.

What this unlocks — and when it's worth it

Some tasks genuinely can't be written as workflows because you don't know what the steps will be until you start. Those are the agent cases:

  • Investigation tasks — "find out what happened with X" where the trail leads somewhere you can't predict.
  • Research tasks with branching — competitor analysis where a pricing page leads to a blog post leads to a press release leads to an investor update.
  • Planning tasks with options — "given these constraints, propose three approaches and pick one."
  • Open-ended questions — anything where the right answer requires looking something up, then something else, then deciding which of those two things matters more.

For all of these, a workflow can't work — because you'd have to write out every possible branch in advance, which defeats the point. An agent handles the branching for you.

◆ The 2026 reality

Demos work. Production often doesn't.

Agents demo beautifully. A well-built agent looks intelligent — reasoning, using tools, producing a real report at the end. The problem is what happens on the 47th run, with a weird input, when the session memory has filled up with half-useful context, and the agent decides to call the same tool twenty times in a row because it can't quite tell if it's done.

Industry data from 2026: agent architectures can cost 10× more than the equivalent workflow (more reasoning calls, more tool invocations, more tokens). Debug time is also roughly 10× higher, because a workflow has one path to check and an agent has every path it could have taken. Neither is a reason to avoid agents — but it's why Module 3 teaches you to build them conservatively, with tight guardrails and full observability.

One honest framing Agents don't really "think." They predict. A large language model generates text based on patterns; an agent wraps that prediction in a loop with tools. The loop creates the appearance of reasoning, and in practice it often produces reasoning-like results — but it helps to remember it's still prediction underneath. That's why prompts and tool descriptions matter so much: they shape what the agent predicts it should do.

The craft-first principle for agents

Here's the rule every careful builder converges on: start with a workflow. Replace a single step with an agent only when the path through that step can't be written down in advance.

So a good agent deployment often looks like this:

  • A workflow triggers on a schedule, pulls known data, and prepares context — deterministic, cheap, reliable.
  • An agent step inside that workflow does the genuinely intelligent bit — research, synthesis, judgement — where the path varies.
  • A workflow takes the agent's output, validates it, logs it, and fans out to downstream actions — back to deterministic behaviour for the operational parts.

This is the pattern Build #2 follows. A scheduled workflow fires every Monday, an agent does the actual research, a workflow formats the findings and writes them to a Google Sheet. Agents for the fuzzy middle. Workflows for the predictable edges.

The one thing to remember

Workflows follow instructions. Agents make choices. That's where the power and the risk both come from.

Three ingredients — tools, memory, and a decision loop — turn a prompt into an agent. Everything else in Module 3 is about building that carefully, watching what it chooses, and giving yourself a way to stop it when it chooses wrong.

◆ Find your candidate

Name one task in your work where the path isn't fixed

Before Lesson 3.2's build, identify a real task in your business where you genuinely can't write the steps in advance. That's your candidate for when you'd reach for an agent of your own after Build #2.

Pick one candidate. Write down: - Task: (one sentence) - Why a workflow wouldn't work: (what varies between runs?) - What tools would an agent need? (web search? URL reader? database lookup? email? calendar?) - What's the cost of it getting something wrong? (low-stakes background research? customer-facing? something that costs real money?) Examples to get you started: - Weekly competitor research (what's new on their sites?) - Inbound lead investigation (who is this person? what's their company about?) - Research synthesis (summarise these 12 articles, find the three most important ideas across them)

Keep the answer somewhere you can find it. After Build #2 is running, that candidate becomes the template for your next agent.
