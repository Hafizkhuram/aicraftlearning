Making it bulletproof — error handling and logging

A workflow that works on your clean test data isn't finished. It's finished when it survives Monday morning — a bad email, a Gmail API hiccup, an expired credential at 3am, and you still know what happened.

This is the Module 2 closer. No new build — same Build #1 from Lesson 2.2, now hardened. Four small additions turn a working workflow into a production-ready one: per-node retries, a dedicated error-handler workflow, a Google Sheet log of every run, and a simple kill switch.

By the end of this lesson, Build #1 won't just run. It'll keep running, tell you when it doesn't, and let you stop it in one click if you need to.

!
The common misconception

"It's been running fine for a week — it's done." Running fine so far is different from running well when something unusual happens. The unusual is the point of this lesson. Every production system eventually hits a weird input, a rate limit, a credential expiry. If your workflow fails silently when that happens, you won't know until a customer tells you. The four additions here make silent failure impossible.

What changes

Before the additions, Build #1 has one failure mode that matters: the whole thing just stops. A node errors, the execution goes red in the log, and unless you happen to look at the Executions tab you never know. After the additions, the workflow has four layers of defence, and every run — success or failure — writes a line you can scan.

Before

Build #1 as shipped

  • One failure = silent stop
  • No retry on transient errors
  • No notification when it breaks
  • No idea how much it's costing
  • No quick way to pause it
After

Build #1, hardened

  • Transient errors auto-retry 3×
  • Real failures trigger an alert
  • Every run logs to a Sheet with cost
  • Kill switch flips in five seconds
  • You find out within minutes, not days

Layer 1 — Retries for transient errors

Most "errors" aren't real errors. They're momentary hiccups — a Gmail API timing out for a second, Claude's endpoint hitting a brief rate limit, a flaky network packet. None of those need your attention. They just need the node to try again.

n8n has a per-node Retry on Fail setting. Toggle it on, pick a retry count, and the node handles transient failures silently. This is the cheapest layer of defence — two clicks, and maybe half your "failures" disappear.

Enable retries on your three external-service nodes

Open Build #1. For each of these nodes — the Gmail Trigger, both Claude nodes, and the Gmail Create Draft / Add Label actions — do this:

  • Open the node
  • Click the gear icon (top right of the node panel) → Settings
  • Toggle Retry on Fail on
  • Set Max Tries to 3
  • Set Wait Between Tries to 5000 ms (5 seconds — good for rate-limited APIs like Claude)
Screenshot · Retry on Fail settings
Screenshot to add: a node's Settings panel with Retry on Fail toggled on, Max Tries set to 3, Wait Between Tries set to 5000.
Why 5 seconds, not the default 1 second? Rate-limited APIs (Claude, OpenAI) often want you to back off for several seconds before retrying. Hitting them again 1 second later just gets rate-limited again. 5 seconds is the sweet spot — long enough to let the rate-limit window pass, short enough that users don't notice the delay.

Layer 2 — An error-handler workflow

Retries catch the small stuff. For everything else — expired credentials, malformed data, real outages — you want a separate workflow that runs only when Build #1 fails. This is n8n's Error Trigger pattern, and it's how you stop being the last to know.

Create a new workflow called "Error Handler"

Workflows → Add Workflow → name it Error Handler. The first node will be special — an Error Trigger.

Add node
Search Error Trigger — pick it as the workflow's first node
Purpose
This node fires whenever a linked workflow errors, receiving the error details automatically

The Error Trigger doesn't need configuring beyond just being there. It auto-receives a payload describing what broke — the workflow name, the execution URL, the exact error message, the node that failed.

Add a notification action

After the Error Trigger, add a Gmail Send Message node (or Slack if you prefer). Send yourself a compact summary of what broke.

That link is the important bit. One click and you're looking at the exact failed execution in the Executions tab — the Lesson 2.3 debugging loop, triggered by a problem instead of you stumbling on it.

Wire the Error Handler to Build #1

Go back to Build #1 (the email triage workflow). Open Workflow Settings (bottom-left menu or the three-dot button).

Error workflow
Select Error Handler from the dropdown

Save. From now on, any real failure in Build #1 automatically fires the Error Handler, which emails you within seconds.

One honest limitation: Error Workflows don't fire if the failure is in the trigger node itself (e.g. Gmail OAuth expires). For those, the workflow simply stops polling and you only notice in the Executions tab — which is why checking the log weekly (Lesson 2.3) still matters.

Layer 3 — Logging every run to a Sheet

The cost_gbp formula. For Claude Sonnet 4.6 at 2026 rates, roughly £0.0025 per 1K input tokens + £0.0125 per 1K output tokens. The expression: {{ (($('Message a Model').item.json.usage.input_tokens / 1000) * 0.0025) + (($('Message a Model').item.json.usage.output_tokens / 1000) * 0.0125) }}. Rough but good enough — it'll tell you whether you're spending £2/month or £20/month.

Test it — trigger a run and watch the row appear

Send yourself a test email. Wait a minute. Open the log sheet. A new row should appear with the timestamp, classification, token counts, and cost. If something's blank, click the failed node in the execution log — it's almost always an expression pointing at the wrong node name.

After a week, this sheet becomes genuinely interesting. Pivot by category. Sum the cost column. Count drafts created versus not. Most "dashboards" you'd pay for are just this sheet with nicer fonts.

Layer 4 — The kill switch

Every production workflow needs a way to stop it in a hurry, without unwiring anything. A misbehaving classifier on Monday morning should be a five-second fix, not a ten-minute scramble through n8n settings.

The cleanest pattern is a single cell in a Google Sheet that the workflow checks at the start of every run. If the cell says active, the workflow proceeds. If it says anything else — paused, off, stop — the workflow stops immediately, logs "skipped" to the run log, and does nothing.

The n8n-native option

Just toggle Inactive. Fastest if you're near your laptop. One click on the canvas.

The portable option

A kill-switch cell. Works from your phone, your team's phone, anywhere with Google Sheets. Slower to build, faster in a panic.

Add a "killswitch" tab to the log sheet

In the same Build #1 — run log spreadsheet, add a new tab called killswitch. Cell A1: status. Cell A2: active.

That cell is the switch. Type paused and hit enter — the workflow stops processing within one poll cycle (up to 60s).

Insert a check at the start of Build #1

Right after the Gmail Trigger, insert two new nodes:

  1. A Google Sheets Get Row(s) node — reads cell A2 from the killswitch tab.
  2. An IF node — condition: does the cell equal active?

The IF's true branch continues the workflow as before. The false branch goes straight to the logging step with classification: skipped — so you still have a record that an email arrived during the paused window, just no classification or draft.

Test both states

Send yourself a test email with the switch set to active. Confirm normal behaviour — classified, labelled, possibly drafted.

Change the cell to paused. Send another test email. Wait 90 seconds. The log should show a skipped row, and no label or draft should have been created. Switch back to active.

You now have a portable, auditable, five-second emergency stop.

The severity principle. Not every log line deserves a notification. Classifier "skipped" (because of kill switch) is informational — worth logging, not worth pinging you about. A real node failure is critical — worth a notification. Separating these prevents alert fatigue, which is the thing that quietly kills every monitoring system within a month.
The one thing to remember

Logs, error paths, kill switch. The three things that separate a toy from a tool.

Retries catch the noise, error handlers catch the real failures, logs tell you what actually happened, and the kill switch lets you stop everything without panicking. Every future build in this course gets the same four layers.

The Monday-morning test

Before moving on, one last exercise that proves your hardening actually works. This is the cheapest way to catch a defect in your error handling before it catches you.

Turn Build #1 off for a day. Turn it back on. Verify no emails were missed.

Specifically:

  • On a Friday evening, flip the kill switch to paused (or toggle the whole workflow Inactive)
  • Over the weekend, send yourself a few test emails at random times
  • Monday morning, flip the switch back to active
  • Check the run log and your inbox. Did the weekend emails get picked up? Were labels applied correctly? Did the log record the skipped runs during the paused window?

If yes, you have a genuine production-ready workflow. If something's off — emails that should have been processed weren't, or labels came out wrong — that's cheap knowledge, found in a safe exercise rather than the day a real outage happens.
