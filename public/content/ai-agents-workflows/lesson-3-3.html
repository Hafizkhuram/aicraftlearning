Giving agents memory and context

Your agent just ran for the second week. It reported the same five findings as last week. Let's fix the reason — and turn a repetitive assistant into one that actually tracks change over time.

Build #2 works, but it has one invisible flaw: it forgets. Every Monday it starts completely fresh, with no idea what it told you the week before. Three weeks in, you're reading the same "Acme launched Widget Pro 3" summary for the third time. This lesson fixes that with a small, surgical change — no rebuild.

!
The common misconception

"Agents remember what they did last time — they're intelligent, aren't they?" They're not, and they don't. An LLM at the heart of your agent has no memory at all between runs. It sees whatever you put in its input this time, and nothing else. The Window Buffer Memory from Build #2 only lasts within a single session. For memory across runs, you have to feed the agent its own past as input. Literally.

Two kinds of memory

Module 3 uses the word "memory" for two quite different things. Keeping them separate saves you hours of confusion later.

Session memory

What Build #2 already has

The Window Buffer Memory sub-node from Lesson 3.2. It lets the agent remember what it did five minutes ago — "I already checked Acme's homepage, try the blog next." Essential for multi-step reasoning inside a single run.

Scope: this single agent run.

Long-term memory

What we're adding today

The ability to see what the agent did last time it ran — last Monday, or the Monday before. We fake this by looking up past outputs (already in the competitive_intel Sheet) and feeding them to the agent as input.

Scope: this week's run, plus recent weeks' context.

Session memory lives inside the agent. Long-term memory lives outside it, in something the agent can read — a Sheet, a database, a text file. This lesson teaches long-term. The trick is unglamorous: the agent doesn't really remember anything. We just give it the information it needs, every time it runs.

The pattern — retrieval before reasoning

The fix has a shape that's worth naming, because you'll use it every time you want an agent to seem like it remembers:

Before the agent reasons, retrieve what it already knows. Pass that as context. Let the agent decide what's new.

In Build #2, that means: before the agent researches the three competitors this week, we read the last few weeks of rows from the competitive_intel Sheet. Then we pass the whole list of "here's what you reported recently" as input alongside the "here's this week's task." The agent now has everything it needs to decide "this is new" versus "I already said that a fortnight ago."

The surgery — three small changes

Open Build #2. Three changes will take about ten minutes.

Insert a Google Sheets "Get Rows" node before the AI Agent

Between the Competitor list (Edit Fields) node and the AI Agent node, add a Google Sheets node.

Credential
Your existing Google Sheets credential.
Resource
Sheet Within Document
Operation
Get Row(s)
Document
Your competitive_intel sheet (the same one Build #2 already writes to)
Sheet
Sheet1
Filters
Leave empty — we want all recent rows, not filtered by competitor. The agent does that sorting.

Under Options, add one more setting:

When Filter Has Multiple Matches
Return All Matches

You'll also want to cap the pull — you don't need months of history, just enough to spot repetition. The simplest way: sort your sheet by date descending, and have this node read the latest rows only.

For small sheets this is fine as-is. With fewer than 50 total rows (a few months of weekly runs on three competitors), pulling everything is trivially cheap. Optimise only if your sheet grows past a few hundred rows — at which point, add a date filter in Options to grab the last 6-8 weeks only.

Update the AI Agent's input expression

Open the AI Agent node. Find the Prompt (User Message) field — the expression you set in Lesson 3.2.

Replace it with this:

Updated user-message expression Paste into Prompt field
# This week's research task Research these competitors this week: {{ $('Edit Fields').item.json.competitors }} # Recent findings (past few weeks) — don't repeat these {{ $('Google Sheets').all().map(r => `${r.json.date} | ${r.json.competitor} | ${r.json.summary}`).join('\n') }}

The second half is the new bit. The $().all().map() pattern pulls every row the Sheets node returned and formats them as a clean list — one line per past entry — so the agent reads them naturally as context.

Update the system prompt — one new rule

Open the System Message field. Find the # Rules section. Add this line:

New rule to add Append to # Rules section
- Before writing a summary for a competitor, review the "Recent findings" list in your input. If this week's finding is something you already reported in the last 4 weeks, say so explicitly ("No notable change since last week's report on Widget Pro 3") — don't repeat the same summary. Focus on what's genuinely new or changed.

Save the workflow. That's it. Three changes, ten minutes.

Same agent, same tools, same system prompt minus one line. The only structural change is the input it sees — and the behaviour transforms. That's the leverage of context done well.

One honest footnote What you've just built is a simple form of RAG — retrieval-augmented generation. In technical writing, "RAG" usually refers to fancier setups involving vector databases and semantic search over large document collections. For most business agents, what we did here — look up recent rows from a Sheet, pass them as context — is 80% of the value for 5% of the complexity. You can go deeper with vector stores when a Sheet genuinely stops being enough. Until then, this is the right ceiling.

Where this pattern unlocks more

Retrieval-before-reasoning is reusable. The same architecture works for every situation where an agent needs to "know what's happened recently":

  • Customer support agent — before replying to a new ticket, look up that customer's previous interactions from a Sheet or CRM and pass them as context. The agent won't re-ask questions they've already answered.
  • Sales research agent — before briefing you on an inbound lead, look up whether their company has been researched before. The agent skips repeating the stable facts and focuses on what's new.
  • Weekly finance summary — before reporting this week's numbers, look up the last four weeks' reports. The agent notices trends instead of just repeating the current snapshot.

The ingredient list is always the same: a place to store past outputs (usually a Sheet, sometimes a database), a Get Rows node before the agent, and one line in the system prompt telling the agent to actually read the history. Three changes, works everywhere.

The one thing to remember

Every agent needs somewhere to look up its own past. Without it, it repeats itself forever.

"Memory" for an agent is usually just reading its own previous outputs back into its input. Unglamorous, cheap, and it's the difference between a novelty and a tool.
