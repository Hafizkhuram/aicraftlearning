Building AI Agents & Workflows Beyond building

Build #3 — connecting AI to your real tools

Your meetings produce action items. Most of them die in your notebook. By the end of this lesson, they live in your project tool instead — and you'll have a pattern you can reuse for every tool your business actually runs on.

This is Build #3, and it's deliberately the simplest of the three. Build #1 taught you the workflow shape. Build #2 taught you the agent shape. Build #3 shows you the reusable pattern — the four-stage shape that connects AI to any tool with an API. Once you've built it once, the second one takes twenty minutes.

!
The common misconception

"I need a different automation for each tool — one for Notion, a different one for Asana, a different one for Slack." You don't. They're all the same shape. The trigger might change. The notify step might change. But the pattern in the middle — something happens, AI extracts structure, route the result somewhere, notify a human — is identical. Learning it once is learning it for everything.

What you're building

Build #3 spec

Meeting transcript → action items → your project tool

  • When: every time a new file appears in a specific Google Drive folder (where your meeting transcripts land).
  • Given: a transcript as plain text (most meeting tools now export text transcripts automatically).
  • Claude extracts structured action items — each with a task description, an owner, and a due date — as JSON.
  • For each action item, create a new page in a Notion database (or a task in Asana, or a message in Slack — same pattern, one-node swap).
  • Email the meeting organiser a summary of what was captured.

One important framing before we start: this is a workflow, not an agent. The path is fixed — every transcript goes through the same five steps. No decision loop is needed. This is Lesson 3.1's craft-first principle at work: use a workflow where the path is predictable, agents only where it varies.

The pattern — four stages

Before we build, look at the shape. This is the pattern. Memorise it.

Four stages. Each one is a node or two in n8n. Swap the trigger for "new Slack message" and the notify for "text me" and you have inbound-lead processing. Swap the trigger for "new webhook from our billing system" and the route for "create a support ticket" and you have invoice-problem handling. It's the same build every time.

Stage 1 — The trigger

Create the Drive folder, then the workflow

In Google Drive, create a folder called meeting-transcripts. This is where transcripts will land — either dropped in manually, or (more realistically in 2026) exported automatically by your meeting tool.

In n8n, create a new workflow called Meeting action items. Add a Google Drive Trigger as the first node.

Stage 2 — The extraction

Now the heart of the build. Claude reads the transcript and returns structured JSON. Structure matters here — the downstream nodes will iterate over the extracted items, so the output has to be parseable, not just readable.

Add a Claude node configured for structured output

Add an Anthropic node (the same node type you used for Build #1's classifier). Configure:

For the message content, include the prompt below and the transcript. Pull the transcript text using: {{ $binary.data.toString('utf-8') }} — this converts the binary file contents to a readable string.

Build #3 extraction prompt Paste into Message field
# Role You are an action-item extractor for meeting transcripts. # Task Read the transcript below. Identify every action item — any concrete task that was agreed to, assigned, or committed to during the meeting. Ignore vague intentions ("we should think about…"). Focus on specific commitments. For each action item, extract: - task: a clear, specific description (1 sentence, imperative mood — "Send the Q3 forecast to Maya" not "Someone should probably send…") - owner: the person responsible, as their first name. If unclear from the transcript, use "Unassigned". - due_date: ISO format (YYYY-MM-DD) if stated or implied. If not specified, use null. - context: one brief sentence of why this task exists (helpful when the owner sees it later). # Output format Return ONLY a JSON array. No preamble, no explanation, no markdown fences. Just the array. If no action items exist, return an empty array: [] # Transcript {{ $binary.data.toString('utf-8') }}

Three things about this prompt that matter:

  • "Focus on specific commitments." Transcripts are full of hedging ("we should probably think about the dashboard at some point"). Without this instruction, Claude will dutifully extract vague items and clog your project tool with noise.
  • "Imperative mood" — a small craft detail that makes the output feel like real tasks. "Send the forecast to Maya" reads like work; "Discussion of sending the forecast" reads like a meeting note.
  • "Return ONLY a JSON array. No preamble." This is the critical line. Without it, Claude often wraps JSON in markdown code fences or adds explanation — which breaks the downstream parsing. Be explicit.

A well-extracted output looks like this:

Claude's structured output
[ { "task": "Send the Q3 forecast to Maya", "owner": "Sam", "due_date": "2026-04-30", "context": "Maya needs it for the board deck." }, { "task": "Review the new pricing deck and flag concerns", "owner": "Priya", "due_date": "2026-05-02", "context": "Decision needed before next week's customer meeting." } ]

Stage 3 — The route

Claude returns one blob of JSON containing multiple items. We need to fan that out — one Notion page per action item. n8n has exactly the right node for this.

Parse and split the JSON array

Add a Code node (or Set node with expressions) right after Claude. Its only job: take Claude's text response, parse it as JSON, and output each action item as a separate n8n item — so subsequent nodes run once per item.

Language
JavaScript
Mode
Run Once for All Items
Code
const text = $input.first().json.content[0].text;
const items = JSON.parse(text);
return items.map(item => ({ json: item }));

After this node runs, n8n shows you N separate items flowing downstream — one per action item Claude extracted. Each subsequent node runs once per item automatically.

If you're wary of code nodes: n8n's AI-native alternative is the Split Out node, which does the same thing with no JavaScript. Either works. The code version is slightly more defensive because it can handle Claude's occasional extra whitespace or fence marks.

Create a Notion database page for each item

In Notion, create a database called Action items with these properties: Task (title), Owner (text), Due (date), Context (text), Source (text — for the meeting reference). Critical step: open the database → Share → add your n8n integration. Without this, n8n can't write to it even though OAuth says you're connected.

Stage 4 — The notify

After every item has been routed, one summary email to the meeting organiser — so they know the transcript was processed and see a summary of what was captured. Same Gmail node you used in Build #1.

Aggregate and email the summary

After the Notion node, add an Aggregate node — its job is to merge the N separate items back into one output, so the email step runs once rather than N times.

One click in the email takes the meeting organiser to Notion to review.

Test it with a real transcript

Activate the workflow. Drop a real meeting transcript into the meeting-transcripts folder — anything from 2-5 minutes of conversation is enough. Within 5 minutes, you should see:

  • The Drive Trigger fires (visible in Executions)
  • Claude extracts the items (inspect the Anthropic node output to see the raw JSON)
  • One Notion page created per action item
  • One summary email in your inbox

If any node throws an error, the Error Handler workflow from Lesson 2.4 catches it — assuming you wired this workflow to the same handler in its Settings.

When Claude's extraction is off. Read the raw output from the Anthropic node. If it's returning JSON wrapped in markdown fences, tighten the "no preamble" line. If it's extracting vague items, add a concrete example of "don't extract this" to the prompt. If it's missing the owner, add an example of handling ambiguous transcripts. Three small prompt tweaks handle 90% of extraction errors — same pattern as Lesson 2.3's debugging loop.

Same pattern, different tool

Here's the point of the whole lesson. The route and notify stages are the only bits that change between tools. Everything else stays the same. Need Asana instead of Notion? Swap one node. Need Slack? Swap one node.

Notion canonical
Node: Notion → Create Database Page

Best when the team already lives in Notion. Database pages with rich properties (owner, due date, status).

Asana swap
Node: Asana → Create Task

Best for teams managing delivery. Tasks carry assignees, due dates, and project tags natively.

Slack swap
Node: Slack → Send Message

Best for lightweight tracking. Post action items to a channel; owners react with ✓ when done.

All three are one-node substitutions. The trigger, the extraction, the code-splitting, the aggregate, the notification email — all identical. This is why the pattern is the valuable thing, not the specific build. Once you see it, every tool in your business looks like a potential destination.

Notion's explicit-share gotcha. Unlike most n8n integrations, OAuth alone isn't enough for Notion. You have to open the target database in Notion's UI, click Share, and add your n8n integration by name. If your node shows an empty database dropdown or "not found" errors, this is almost always the reason. Annoying but specific to Notion.
The one thing to remember

The pattern is always the same: trigger, extract, route, notify. Once you've built one, you've built them all.

Most of the "AI automation" ideas you'll think of over the next year are just this shape with different parts plugged in. Your stack is now extensible — every tool with an API is a potential destination, and the wiring is the same every time.
