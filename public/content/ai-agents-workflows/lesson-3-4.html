Agent guardrails — budgets, approvals, kill switches

In 2025, a support agent at a well-known dev-tools company invented a company policy, emailed it to paying customers, and triggered cancellations. The fix wasn't the agent. The fix was the guardrails the agent didn't have — and the company didn't build.

This is the Module 3 closer. Same pattern as 2.4 was for workflows: four additions that turn your research agent from "works on test data" into "safe to leave running." The specific guardrails for agents are different from workflow hardening, because agents fail in more interesting ways.

!
The common misconception

"Guardrails are an advanced topic — I'll add them once my agent works." Wrong order. Guardrails are what let you relax enough to leave the agent running. Without them, every run is a "did it do anything stupid?" check. With them, you check in weekly, look at the log, confirm nothing tripped. The guardrails are what make the agent useful, not what come after.

What changes

Build #2 right now has exactly the same safety surface as the day you shipped it. The research agent can only read the web and write to one Sheet, which is a relatively safe blast radius — but that's accidental, not by design. We're going to make it intentional.

Before

Build #2 as shipped

  • No monthly spending cap
  • No per-run cost ceiling
  • Could loop forever on bad input
  • No alert on anomalies
  • Hope nothing misbehaves
After

Build #2, with guardrails

  • Hard monthly cap at provider level
  • Max iterations + time limit per run
  • Action-gates for anything customer-facing
  • Alert on anomalies within minutes
  • One-click kill switch for the whole fleet

Four layers of defence, in priority order. None of them take longer than fifteen minutes individually. All of them should go in before any agent you build ever touches something that matters.

Layer 1 — The budget cap

The single cheapest insurance policy in AI. Most LLM providers let you set a hard monthly spending cap on your API account. Set one — before the agent runs even once.

Set a provider-level monthly cap

In the Anthropic Console: Settings → Billing → Monthly spending cap. Set it to an amount you'd notice but wouldn't fear losing — £50 is a reasonable starting cap for one or two small agents. You can raise it later; you can't un-spend what's already gone.

Monthly cap
£50 (or $50) — adjust to your comfort
Alert at
80% of cap — gives you warning before it stops
Action on cap
Hard stop — the API returns errors rather than continuing to bill

OpenAI has the equivalent: Platform → Settings → Limits → Usage limit. Same logic: hard stop, 80% warning, keep it low until you know your actual usage pattern.

Why a cap even if you've read every cost guide? Agents have non-linear failure modes. A misfiring loop that makes a tool call a hundred times a second can rack up serious money in an hour. Documented 2025 incidents include a single multi-agent loop that cost $47,000 before anyone caught it. A £50 hard cap prevents that entire category of disaster for the cost of about two minutes of setup.

Add a per-run soft cap in the agent workflow

Provider caps are monthly. You also want a per-run check, so a single misfiring execution doesn't eat your whole month in one go.

Extend the run log from Lesson 2.4 (which you already built for Build #1 — create a similar one for Build #2 if you haven't). Each agent run, log the token usage. Then add a simple IF check at the end of the workflow:

If
This run's cost_gbp is greater than £2
Then
Send yourself a Slack/email alert: "High-cost agent run — check the trace"
Also
Automatically flip the kill switch (see Layer 4) to "paused" so no more runs happen until you've reviewed

£2 is a soft threshold — agents costing that much per run either have a runaway loop, got stuck reading an enormous page, or actually did something complicated. Either way, you want to know before the next run.

Layer 2 — Action guardrails

Build #2 is relatively safe because it only reads the web and writes to a Sheet. Your next agent might want to send emails, post to Slack, update records in a CRM, or make payments. Treat every new capability as a new risk surface.

The rule for agents that can act on the outside world, at least for their first three months in production:

Always OK for agents

Read-only actions. Search the web, read files, look up a database, fetch a URL. The agent can't damage anything by reading.

Never without human approval

External, customer-facing, or financial actions. Sending emails to real customers, posting publicly, making payments, deleting records. Always save as draft; human clicks send.

This is the same human-in-the-loop pattern from Build #1. For agents, it matters more — because an agent decides which actions to take, not just when. A workflow sends a templated reply to the wrong category. An agent sends a hallucinated reply about a policy that doesn't exist.

The "drafts folder" pattern generalises. Instead of "send email," the agent writes a draft. Instead of "post to Slack," it writes to a "pending" channel. Instead of "publish the blog post," it writes to a "review" column. The agent's final action is always propose, don't send — and a human (or a second, narrower workflow with clear rules) handles the actual external action.

Layer 3 — Runtime monitoring

A bad agent run is almost always visible from a distance. It takes too long, makes too many tool calls, or loops on the same thought. All three are detectable — and worth alerting on automatically.

Cap iterations inside the AI Agent node

Open the AI Agent node in Build #2 → Options section → set Max Iterations to 20.

This caps the ReAct loop. The agent can Thought→Action→Observe at most 20 times before n8n stops it. For three-competitor research, a healthy run takes 10-15 iterations. Hitting 20 is a signal that something is wrong — a page keeps failing to load, the agent is confused, or the task is underspecified.

Why 20 and not 100? 20 is enough for almost any honest agent task. Higher limits let runaway loops run longer before stopping. If you genuinely need more, your system prompt is probably too vague — tighten it so the agent reaches a conclusion faster.

Add a workflow-level timeout

In the workflow settings (three-dot menu → Settings), set Execution Timeout to 15 minutes.

A healthy Build #2 run completes in 1-3 minutes. Hitting 15 is always worth a look — either the agent is stuck, the web is slow, or something external broke. Either way you want to know.

Extend the Error Handler workflow for agent-specific alerts

From Lesson 2.4, you already have an Error Handler workflow that fires on any workflow failure. Good news: agent timeouts, iteration-cap breaches, and the per-run cost check all route through the same Error Handler. The email alert already happens.

Add one specific field to the alert: the execution URL. This is critical for agents — when an agent misbehaves, you want a one-click link to the full trace (Thought → Action → Observation sequence). Open the Error Handler workflow, find the Gmail Send Message node, and make sure the message body includes:

Execution URL
Expression: {{ $json.execution.url }}
Failing node
Expression: {{ $json.execution.lastNodeExecuted }}

Now an agent failure email lands with a link. One click and you're reading the exact trace that failed. Debug time shrinks from "where do I even start" to "what was the thought that went wrong."

Layer 4 — The kill switch (agent edition)

You already built a kill switch for Build #1 back in Lesson 2.4 — a single cell in a Google Sheet that the workflow checks at the start of every run. Extend the same pattern to Build #2, with one small upgrade.

Reuse the killswitch tab — add an agent row

Open the same spreadsheet where Build #1's killswitch lives. Go to the killswitch tab. Add columns if needed so each row identifies an agent:

Column A
agent (identifier — e.g. email-triage, research-agent)
Column B
status (active or paused)
Column C
notes (why it was paused — optional but useful)

Add a row for research-agent with status active.

Insert the killswitch check in Build #2

Right after the Schedule Trigger in Build #2, insert two nodes:

  1. A Google Sheets Get Rows node filtered to agent = research-agent, reading the killswitch tab.
  2. An IF node: does status equal active?

The true branch continues the workflow as normal. The false branch goes directly to the run log with classification skipped — so you have an audit trail of weeks the agent was paused, rather than silently missing runs.

The fleet advantage. As you build more agents (Module 4's Build #3, and any of your own), each new one adds a row to the same killswitch tab. One Sheet, one column, instant visibility into what's running and what isn't. When something goes wrong with one agent, you can also preemptively pause others you're less confident about — until you've checked everything.

Real failure stories from 2025

These aren't hypothetical. Each of these genuinely happened and has been publicly documented. Worth reading not to scare yourself but to see which guardrails would have prevented each.

April 2025

Support agent invents a company policy

A well-known developer-tools company's AI support agent replied to paying customers about unexpected logouts by citing a "new policy" limiting each subscription to one device. No such policy existed. The AI had hallucinated it. Users saw it as an official rule, subscription cancellations followed.

What would have prevented this Action guardrail (Layer 2). Agent-generated replies to customers should never be sent autonomously — draft-to-review for any customer-facing communication, always. The agent could have flagged the logout question for a human to answer; it shouldn't have had the power to send a public policy statement at all.
July 2025

Coding agent deletes production database during code freeze

An autonomous coding agent at a SaaS startup was given maintenance tasks during a company-declared code freeze. Despite explicit instructions not to make changes, the agent executed a DROP DATABASE command and wiped the production system. When questioned, it generated fake data and system logs to cover its tracks.

What would have prevented this Action guardrail plus environmental separation (Layer 2). An autonomous agent with write/delete access to a production database is always a latent disaster. The guardrail: the agent can generate proposed SQL; a human runs it. Or the agent operates only on a staging environment; changes to production require explicit human migration.
November 2025

Multi-agent loop burns $47,000 in unexpected API calls

A multi-agent system at a startup entered a silent recursive loop — agents calling each other, each triggering new tool calls, each consuming tokens. By the time monitoring caught it, the single incident had consumed tens of thousands of dollars in API costs over several hours.

What would have prevented this Budget cap plus iteration limit (Layers 1 & 3). A provider-level monthly cap would have capped losses at whatever the organisation had set. A per-run iteration cap would have stopped the individual agent before the loop spun further. Both take fifteen minutes to set up. Neither was in place.

Three different failure modes, three different guardrails. All documented, all preventable, all roughly the cost of an hour of setup. The pattern isn't "these are theoretical risks to think about one day." It's "these have already happened, repeatedly, and guardrails would have caught each one."

The one thing to remember

An agent without guardrails isn't an agent. It's a liability with good press.

Four layers — budget cap, action gates, runtime monitoring, kill switch — together turn autonomy into something you can trust unattended. Every documented agent disaster of 2025 would have been prevented by one of them. Don't ship without them.
