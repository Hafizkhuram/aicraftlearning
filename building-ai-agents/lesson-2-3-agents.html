Debugging — when things go wrong

Your first workflow is going to break. Good. That's where the real learning starts — and where every experienced builder becomes faster at this than the beginner who panics.

By the end of this lesson you'll have a short, reliable diagnostic process that handles 90% of the things that will go wrong with Build #1 — and every build after it. No panic. No guessing. Just read the log, isolate the failing node, fix the smallest thing.

!
The common misconception

"If my workflow isn't working, I must have set something up wrong from the start." Usually not. Real workflows break because the world is messy — unexpected email formats, temporary API hiccups, credential expiry, rate limits. Debugging isn't undoing mistakes. It's the normal work of running real systems. Every experienced builder spends more time reading execution logs than writing new nodes.

The four ways it breaks

Almost every failure on Build #1 fits into one of four buckets. The symptom tells you roughly where to look.

Symptom
Nothing triggers at all

Likely cause: credentials, trigger configuration, or workflow is still set to Inactive.

Where to look Open the Gmail Trigger node. Test it manually. If it fetches nothing, the Gmail credential has likely lapsed — re-authenticate. If the credential is fine, check the activation toggle at the top of the canvas.
Symptom
Workflow stops mid-run

Likely cause: API rate limit, credit exhaustion, or unexpected data shape at one node.

Where to look Open the Executions tab. Find the failed run. The red node is the one that broke. Click into it to see the exact error — "rate limited," "insufficient credit," and "cannot read property X of undefined" each mean different things.
Symptom
Classifications are wrong

Likely cause: the prompt isn't specific enough, or the email doesn't match a category.

Where to look Not in n8n. Open the classifier prompt and add an example. If customer questions keep being tagged as spam, add one line: "e.g. 'How do I return an order?' is routine, not spam." Prompts are text you iterate on.
Symptom
Drafts read poorly

Likely cause: the drafting prompt lacks context about your voice, your business, or what "good" looks like.

Where to look The drafting prompt. Add a one-line tone description. Add a sample of how you actually reply. Add the rule "mirror the sender's register — formal if formal, warm if warm." Small changes, big quality jumps.

Notice the pattern. Two of these four live in the prompt, not the workflow. That ratio holds up in practice — most Build #1 failures are prompt problems, not n8n problems.

The executions log is your first stop

The Executions tab in n8n is the single most useful thing in the whole tool for debugging. Most beginners ignore it for weeks and waste hours guessing. Don't be that person.

Screenshot · Executions tab
Screenshot to add: the Executions list showing a mix of green (successful) and red (failed) runs, with timestamps and durations.

Every run of your workflow — successful or failed — appears here. Click any row and you see the workflow as it ran at that moment, with each node's exact input and output. When something breaks, this is always the first place to look. Three things to know:

  • Red border on a node = the failure point. The exact node that errored is visually marked. Click it and the error message appears in plain English.
  • Input and output panels appear on each side of the node. The input is what arrived from the previous step. The output is what this node produced (or would have, if it hadn't errored). Compare the two to see where the data went wrong.
  • Three view modes — Table, JSON, and Schema. Table is friendlier for business data; JSON is the raw truth; Schema shows the shape. If a node is complaining about a missing field, switch to JSON and check the exact key names.

The craft-level move: "Debug in editor"

Here's the trick that separates comfortable debuggers from frustrated ones. When a production workflow fails, you can pull the exact failing data back into your editor in two clicks.

Open the failed execution

Executions tab → click the red row you want to fix.

Click "Debug in editor"

Top-right of the execution view. n8n copies the exact data from the failed run — the specific email, the specific classifier output, everything — and pins it into your editor.

Fix and re-test against the real data

Tweak the prompt, the expression, or the node config. Re-run the workflow. It now runs against the exact data that broke it last time — not synthetic test data you're hoping is representative.

This sounds simple because it is. But it saves you the single biggest time-sink in debugging — trying to reproduce a production bug from memory. Instead, you're working with the exact bytes that broke things.

Pin data while you iterate

When you're tweaking a prompt, you don't want to ping Gmail (or Claude, or Sheets) every time you re-test. That costs API credits, hits rate limits, and is slower than it needs to be.

The fix is pinned data. Run the trigger node once to capture a real email. Click the pin icon on its output. That email is now frozen — every subsequent test run will use the pinned data instead of calling Gmail again.

Screenshot · Pin data icon
Screenshot to add: the OUTPUT panel of the Gmail Trigger with the pin icon highlighted, and the "This data is pinned" banner showing.

Now iterate on your classifier prompt twenty times in a row. The Gmail call happens zero extra times. The Claude call happens once per test. Cost stays pennies. Iteration stays fast.

Pinned data only affects manual testing. Production runs always use fresh data from the actual trigger. You can pin data during development without fear of affecting what happens to real incoming emails.

The debugging mindset

More important than any specific fix: expect errors. Build #1 running cleanly every day for a month would be surprising. Build #1 occasionally tripping on an unusual email, a Gmail API blip, or a Claude rate-limit hiccup — that's just the system being alive.

Three habits separate people who thrive with their workflows from people who abandon them:

  • Check the Executions tab weekly, even if nothing feels wrong. A 5% failure rate you don't notice becomes dozens of missed emails over a year.
  • Read the log before guessing. Every broken workflow has a log entry that tells you what broke. Twenty seconds of reading beats twenty minutes of trial-and-error.
  • Fix the smallest thing. If the classifier is miscategorising, add one line to the prompt — don't rewrite the whole thing. Small changes are reversible; big changes introduce their own bugs.
The one thing to remember

Every broken workflow has an execution log. Read it before you guess.

The log almost always tells you the exact node, the exact input, and the exact error. Guessing is slower and less accurate than reading. Always.

◆ Break it on purpose

Deliberately break your workflow, then fix it

The fastest way to get comfortable with debugging is to do it once on purpose, calmly, while nothing is actually on fire. Ten minutes now saves hours later.

1. Open your Build #1 workflow 2. In the classifier Claude node, deliberately break the prompt — e.g. change the model to an invalid name like "claude-fake-99" 3. Save and run the workflow manually 4. Watch it fail — open the Executions tab 5. Click the red node. Read the exact error message. 6. "Debug in editor" to pull the failed data back 7. Fix the model name 8. Re-run. Confirm success.

You've now seen every step of the debugging loop with zero stakes. Next time it happens for real, you'll know exactly what to do.
