Building AI Agents & Workflows Build your first agent

Build #2 — research agent

By Monday morning, a small machine you built is reading your competitors' websites and writing you a briefing. That's not a demo. That's Build #2 — your first real agent, doing weekly competitive intelligence while you sleep.

This is the big one for Module 3. Build #1 ran a fixed pipeline. Build #2 is a genuine agent — it decides what to look up, which pages to read, and when it's seen enough to stop. The anatomy from Lesson 3.1 becomes a real thing you can inspect, debug, and tune.

Expect this lesson to take 45-60 minutes of hands-on time. Longer than Build #1, mostly because the agent node has more sub-parts and each one matters.

!
The common misconception

"The system prompt is the whole agent — write a good prompt and you're done." Half right. The prompt matters enormously, but so do the tool descriptions. The agent reads both when deciding what to do next, and it ignores tools with vague descriptions. An agent with a brilliant system prompt and three badly-described tools will reason beautifully then refuse to act. Write both carefully.

What you're building

Build #2 spec

Weekly competitive intelligence agent

  • When: every Monday at 7am, automatically.
  • Given: a list of three competitors you care about (their names and website URLs).
  • For each one, the agent investigates recent activity — searches the web for news, reads their homepage and blog, notes what's new.
  • Writes findings as rows in a competitive_intel Google Sheet — one row per competitor per week.
  • Emails you a summary when it's done, so Monday morning you open one email and see the week's competitive landscape.

Architecturally this is the hybrid pattern from Lesson 3.1: a workflow provides the predictable edges (schedule trigger, competitor list, final email), and a single agent node does the fuzzy middle (the actual research).

The agent architecture

n8n's AI Agent node isn't a single node like the ones you've used so far. It's a root node with multiple sub-nodes plugged into it — one for the LLM, one for memory, and one for each tool. Here's the shape:

On the canvas this renders as the AI Agent node in the middle, with five smaller nodes connecting into its underside. The root node runs the reasoning loop; sub-nodes provide capabilities. Change a sub-node and the agent's behaviour changes — swap Claude for GPT, swap SerpAPI for Brave Search, add a calendar tool. The root stays the same.

Why three tools and not ten? Agents get confused by too many tools — 10+ and tool selection starts to misfire. Two to four is the sweet spot for most use cases. We're using three. Resist the temptation to add more until you need them.

Why session memory, not long-term? We want the agent to remember "I've already checked Acme's homepage, now try their blog" within this week's run. We don't need it to remember last week — that data lives in the Sheet already. Simple is better here.

Now the critical bit — the tool description. This is what the agent reads when deciding whether to use this tool. Vague description → agent never searches.

◆ Tool description — paste this into the "Tool Description" field

Use this to search Google for recent news, press releases, product announcements, and blog posts about a specific company. Pass a focused search query like "AcmeWidgets 2026 product launch" or "BetaCo pricing changes recent". Returns structured search results with titles, snippets, and URLs. Use this first to find out what's new; then use the URL reader tool to dig into specific pages.

Notice what this description does: tells the agent when to use it ("recent news, press releases, product announcements"), gives an example query format, and suggests how this tool pairs with the next one. That last sentence is the difference between an agent that searches once and stops versus one that searches, reads, and investigates.

◆ Tool description

Use this to fetch and read the full content of a specific webpage when you have its URL. Best for reading a competitor's homepage, blog post, pricing page, or press release after you've found the URL via a search. Returns the page as plain text. Do NOT use this for Google search queries — use the search tool for those. If a page is very long, focus on the first several thousand characters which usually contain the most important content.

One honest limitation: HTTP Request returns the raw HTML as plain text, which can be huge — some pages are 40,000+ tokens. On very long pages, the agent may only see the top portion. For a first build this is fine; in Module 4 we'll look at trimming the input. Flag it, don't fix it yet.

Step 4 — The system prompt

Now the single most important piece of text in the entire build. The system prompt tells the agent who it is, what it's doing, and how to behave. Open the AI Agent node → System Message field → paste this:

Build #2 system prompt Paste into System Message field
# Role You are a competitive intelligence analyst. Your job is to produce a concise weekly briefing on named competitors — what's new, what's notable, what a founder should know. # Task For each competitor in the input list, do the following: 1. Search the web for recent activity in the last 2-4 weeks (product launches, pricing changes, funding announcements, notable blog posts, press coverage). 2. Read the 1-3 most promising URLs from the search results. Focus on the competitor's own site and credible industry coverage. 3. Summarise the most notable findings in 2-4 sentences. If nothing notable happened, say so — don't invent. 4. Write one row to the competitive_intel sheet using the Google Sheets tool. Include today's date, the competitor name, the summary, and the URLs you used as sources. 5. Move on to the next competitor. # Rules - Work through the competitors one at a time, completely, before moving to the next. - Never invent facts. If the search returns no recent activity, the summary should reflect that honestly. - Keep summaries short and specific. "Launched a new pricing tier on Oct 15" beats "has been making interesting changes." - If a page fails to load or returns garbage, try one alternative URL. Don't loop indefinitely. - When you've written all three competitors to the sheet, output a one-paragraph summary of the overall week and stop. # Output Your final message should be a one-paragraph human-readable summary of the week's competitive landscape, suitable to email to a busy founder.

Two things this prompt does that matter:

  • It defines when to stop. "When you've written all three competitors to the sheet" — agents without a stop condition can loop forever. This one-liner is what turns the ReAct loop into a bounded task.
  • It names the tools by what they do, not what they're called. "Search the web" and "read URLs" — the agent matches these to the tool descriptions you wrote in Step 3. The system prompt and tool descriptions have to speak the same language.

Step 6 — Test it and read the reasoning trace

Hit Test workflow (top right). This runs the whole thing once, now — you don't have to wait until Monday.

The agent will take anywhere from 30 seconds to three minutes, depending on how many pages it reads. Watch it on the canvas — you'll see the nodes light up as the agent calls each tool. Then open the Executions tab and click the run you just made.

This is where you see the full agent trace — the thing Lesson 3.1 promised would be the gold standard of agent observability. Click into the AI Agent node in the execution view and you'll see every turn of the loop:

Read that trace carefully. Every decision the agent made is visible. If a competitor gets a weak summary, you can see exactly why — maybe the search returned nothing useful, maybe the agent read the wrong page. The trace is the single most important thing when debugging agents, and n8n gives it to you for free.

If the agent didn't use a tool you expected. That's almost always the tool description, not the system prompt. Open the tool's node, read its description through the agent's eyes, and ask: "If I'd never seen this system before, would this description tell me when to use this tool?" If no, rewrite it. This one habit fixes 80% of "the agent ignored my tool" problems.
The one thing to remember

An agent is only as good as its system prompt and its tools. Write both carefully; skim neither.

The system prompt defines what to do. Tool descriptions define what can be done. The agent reads both, picks tools it understands, and follows instructions it believes. Weak on either side and the whole thing gets vague.

What to watch for in the first month

Three things to keep an eye on over the first 3-4 weekly runs:

Summary quality. The first week's summary will probably be decent. If weeks 2-3 are worse, look at the trace — the agent is likely searching too narrowly or reading low-quality pages. Tweak the system prompt to be more specific about the kinds of pages worth reading ("prefer the competitor's own site and credible industry coverage over forum posts and aggregator sites").

Cost. A single weekly run of this agent costs roughly 50p-£1.50 in Claude tokens plus ~3-10 SerpAPI searches. Watch the execution time — an agent running for 8+ minutes is usually stuck in a loop; cap it with max iterations (AI Agent node → Options → Max Iterations, set to 20).

The hallucination test. Pick one week's summary. For each claim in it, ask: "Can I find this in one of the source URLs?" If yes, great. If no, that's a hallucination and the prompt needs tightening on the "never invent facts" rule.

◆ Let it run

Before Lesson 3.3, confirm your agent is running weekly

The agent is new territory. The point of this build isn't perfection — it's having a running agent whose reasoning you can read, so Module 3's next two lessons have something real to work with.

Checklist before Lesson 3.3: □ Workflow is activated (Active toggle on) □ First manual test run completed successfully □ You've opened the AI Agent execution and read the full trace □ At least one row has landed in the competitive_intel sheet □ You've received the weekly-briefing email □ You've spotted one thing you'd tune (tool description, system prompt, or competitor list)

That last box matters. No first version of an agent is perfect — noticing what to tune is the first step in owning it.
